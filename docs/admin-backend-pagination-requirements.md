# Admin 列表改造需求确认

## 背景
当前 admin 中以下三个列表虽然有分页 UI，但实际不是完整的真实后端分页：

1. 员工管理
2. 考勤计算
3. 薪资核算

现状是前端拿到整批数据后，再在本地 `slice` 做分页，导致：
- 数据量增大后页面首屏加载变慢
- 分页不是数据库真实分页
- 筛选、总数、翻页行为与真实数据窗口不完全一致

## 本次目标
把以下三个列表全部改成真实后端分页：

1. 员工管理列表
2. 考勤计算列表
3. 薪资核算列表

## 确认口径
### 1. 分页定义
“真实后端分页”指：
- 前端请求时显式传 `page`、`pageSize`
- 后端接口按分页参数查询/裁剪数据窗口
- 接口返回当前页数据与分页元信息
- 前端分页器依据后端返回的 `total/page/pageSize/hasMore` 渲染
- 不再以“整包返回 + 前端 slice”作为主分页实现

### 2. 本次范围
包含：
- 前端 API 参数与返回类型调整
- 三个列表组件的取数、翻页、总数展示、分页状态联动调整
- 后端三个对应接口的分页参数解析与结果返回结构调整
- 必要注释同步
- 必要构建/校验

不包含：
- 无关模块重构
- 列表视觉重设计
- 数据库表结构变更
- 与本次分页无关的业务规则修改

### 3. 三个列表的目标状态
#### 员工管理
- 保留现有搜索/展示语义
- 页面改为按后端分页取数
- 前端不再持有整包员工列表后再本地分页

#### 考勤计算
- 当前筛选条件下按后端返回分页结果
- 翻页时重新请求后端
- 选择日期/月/员工等筛选变化时回到第一页

#### 薪资核算
- 当前筛选条件下按后端返回分页结果
- 翻页时重新请求后端
- 月份/状态/搜索变化时回到第一页

## 验收标准
1. 三个列表都通过后端分页返回当前页数据。
2. 前端翻页会触发新请求，而不是只在本地切片。
3. 切换筛选条件时会回到第一页。
4. 分页总数来自后端，而不是当前页长度。
5. 相关代码构建通过。
6. 如有必要，补充最小验证说明，明确每个列表的请求与返回分页元信息已生效。

## 风险与边界
- 员工管理当前有全局 `employees` 数据被其他模块复用，改分页时不能误伤这些依赖。
- 考勤计算与薪资核算当前有“列表数据即后续弹窗/批量操作源数据”的用法，分页后要保证当前页行为与详情读取不被破坏。
- 本次默认采用“列表分页接口返回 `{ items, total, page, pageSize, hasMore }`”的统一契约，若某页面已有稳定依赖需兼容，需在实现中最小化过渡。

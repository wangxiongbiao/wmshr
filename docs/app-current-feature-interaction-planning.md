# App 当前已有功能交互完善需求拆解

## 目标理解
本轮只针对 app 当前已有功能做界面交互完善，优先把高频功能做成可顺畅使用的闭环；其中考勤记录明确不做详情页，改为直接在列表页完成查看和判断。

## 需要澄清的问题
暂无必须澄清的问题。

## 范围约束
- 只针对 app 当前已有功能的界面交互进行完善。
- 不新增新业务模块。
- 不擅自扩大到 admin 端或后端新能力。
- 优先做功能闭环，不以视觉美化为优先目标。
- 考勤记录不做详情页，列表页直接承担查看与判断职责。

## 模块 / 能力拆解

### 1. 登录与会话交互
已有能力：
- 工号密码登录
- 记住我
- 忘记密码提示
- 自动恢复登录态
- 退出登录

本轮关注：
- 输入校验是否清晰
- 登录中 / 失败 / 成功反馈是否完整
- 会话失效后页面如何引导用户回到可操作状态
- 退出登录确认与结果反馈是否顺畅

### 2. 首页与打卡交互
已有能力：
- 首页展示员工信息
- 读取今日打卡状态
- 上下班打卡
- 定位、地址反查、说明输入
- 成功 / 失败提示

本轮关注：
- 今日状态展示是否足够清楚
- 打卡前、打卡中、打卡后状态是否连贯
- 必填说明的提示是否足够前置
- 定位失败、提交失败、重复点击等异常场景的交互是否闭环
- 删除“每日任务”后首页信息结构是否合理

### 3. 考勤记录交互
已有能力：
- 考勤列表
- 不再保留详情页方向

本轮关注：
- 列表信息是否足够支撑用户判断
- 列表页直接承载查看和判断所需信息
- 空状态、失败状态、异常状态是否清楚
- 列表到首页打卡后的状态回流是否完整

### 4. SOP 交互
已有能力：
- SOP 列表
- 搜索
- SOP 详情
- 已读 / 未读状态
- 确认阅读
- 打开附件

本轮关注：
- 列表与详情的状态衔接是否顺畅
- 已读状态反馈是否及时
- 搜索无结果、加载失败等状态是否完善
- 阅读确认动作的反馈和完成感是否足够

### 5. 我的 / 设置交互
已有能力：
- 我的页面展示员工基本信息
- 进入设置
- 退出登录
- 设置页语言切换

本轮关注：
- 我的页面信息层级是否清晰
- 设置页骨架如何做到最小可用
- 语言切换反馈是否明确
- 当前未完成能力如何避免让用户误以为可用

## 更新后的用户主流程清单

### 流程 1：登录进入 App
入口：
- 打开 App，未登录状态进入登录页

用户动作：
- 输入工号
- 输入密码
- 可选勾选“记住我”
- 点击登录

系统反馈：
- 登录中状态
- 登录成功后进入首页
- 登录失败时给出错误提示
- 忘记密码时可查看“联系管理员重置密码”的提示

结束状态：
- 成功：进入首页并可继续使用
- 失败：停留登录页，用户可修正输入后重试

当前闭环判断：
- 基本闭环已存在
- 后续主要是交互细化，不是功能补建

### 流程 2：进入首页并查看今日打卡状态
入口：
- 登录成功后进入首页
- 或 App 恢复登录态后直接进入首页

用户动作：
- 查看员工名称
- 查看当前时间 / 日期
- 查看今日打卡状态

系统反馈：
- 展示今日是否未打卡 / 已上班打卡 / 已完成打卡
- 根据状态决定后续可执行动作

结束状态：
- 用户明确知道自己当前是否还能打卡、接下来该做什么

当前闭环判断：
- 主路径已存在
- 需要关注状态表达是否足够清晰、是否让用户一眼知道该做什么

### 流程 3：首页执行上班 / 下班打卡
入口：
- 首页已有今日状态
- 用户点击打卡按钮

用户动作：
- 如需说明，输入打卡说明
- 授权定位
- 等待定位和地址解析
- 提交打卡

系统反馈：
- 请求权限中
- 定位中
- 地址解析中
- 提交中
- 成功提示 / 失败提示
- 成功后首页状态刷新

结束状态：
- 成功：今日状态更新为下一阶段
- 失败：用户知道失败原因，并知道是否可以重试

当前闭环判断：
- 这是当前最高优先级主流程
- 已有完整雏形，但最值得做的是交互连续性和异常恢复体验

### 流程 4：查看考勤记录
入口：
- 点击底部“考勤”Tab

用户动作：
- 浏览最近 31 天考勤记录
- 不进入详情页，直接在列表中完成查看和判断

系统反馈：
- 显示每条记录的核心信息
- 允许用户快速看懂某一天是否正常、是否异常、工时如何

结束状态：
- 用户无需再点详情，也能完成日常自查

当前闭环判断：
- 当前列表存在，但原先带有“进入详情页”的路径
- 既然现在明确不要详情页，那么列表页必须承担更多信息表达责任

### 流程 5：查看 SOP 并确认已阅读
入口：
- 点击底部“SOP”Tab

用户动作：
- 浏览 SOP 列表
- 搜索关键词
- 进入 SOP 详情
- 阅读正文 / 查看附件
- 点击“确认已阅读”

系统反馈：
- 列表展示已读 / 未读
- 详情展示内容、版本、更新时间
- 阅读确认后给出成功反馈
- 返回列表后状态应同步更新

结束状态：
- 用户完成 SOP 阅读并完成确认动作
- 列表状态与详情状态保持一致

当前闭环判断：
- 这是当前完成度较高的次主流程
- 重点是状态回流和列表 / 详情一致性

### 流程 6：查看个人信息、设置语言、退出登录
入口：
- 点击底部“我的”Tab

用户动作：
- 查看个人信息
- 进入设置页
- 切换语言
- 或发起退出登录

系统反馈：
- 个人信息展示
- 语言切换生效反馈
- 退出登录确认弹窗
- 退出后返回登录页

结束状态：
- 用户能完成基本账户使用动作
- 不会误以为设置页还有大量已可用功能

当前闭环判断：
- 基本骨架已存在
- 重点是“最小可用感”和“避免误导”

## 考勤列表页替代详情页所需承载的信息项清单

### A. 每条记录至少应让用户一眼看到的信息
1. 日期
2. 上下班时间
3. 工时 / 时长结果
4. 记录类型或状态
5. 备注 / 说明（如已有）

### B. 为替代详情页，列表页应额外承载的判断信息
1. 当天记录是否完整
2. 是否存在异常提示
3. 必要的地点 / 说明摘要
4. 记录间的视觉层级

### C. 列表页必须补齐的交互状态
1. 加载状态
2. 空状态
3. 失败状态
4. 刷新后的反馈一致性
5. 无需点击详情也能完成自查

## 交互缺口清单

### 1. 登录与会话交互缺口
- 登录中是否足够明显，避免重复点击
- 登录失败后是否足够明确告诉用户该改什么
- 恢复登录态失败时是否有清晰回退到登录页的体验
- “忘记密码”更像提示，不像完整支持路径，需要避免用户误解为还能继续操作

### 2. 首页与打卡交互缺口
- 今日状态是否足够“下一步导向”
- 打卡说明必填逻辑需要更前置、更清楚
- 打卡过程步骤较多，用户不一定清楚卡在哪一步
- 打卡成功后首页状态如何回流，需要更强闭环感
- 打卡失败后除了 toast，还需要更明确的恢复指引

### 3. 考勤记录交互缺口
- 列表页承载信息可能不够
- 用户仍被暗示可以点击进入详情
- “查看记录”与“判断记录是否正常”之间还不完全闭环
- 缺少空状态、失败状态、异常状态的清晰表达
- note / 说明 / 地点等信息若有价值，需要直接在列表承担，否则取消详情后信息会丢失

### 4. SOP 交互缺口
- 已读状态是否能及时回流到列表
- 搜索无结果时是否有清晰空状态
- 阅读确认后是否有足够明确的完成反馈
- 附件打开失败时是否有合理提示
- 长内容阅读后的状态变化是否足够明显

### 5. 我的 / 设置交互缺口
- 我的页信息组织是否足够清楚
- 设置页目前偏骨架，需要避免用户误判“通知 / 安全已经可配置”
- 语言切换反馈方式需要清楚
- 退出登录应更像完整的账户收尾动作

## 按依赖顺序排列的任务清单

### 任务 1：确认本轮交互完善边界
目标：
- 明确本轮只做 app 现有功能的交互闭环，不新增业务模块；并确认考勤记录不做详情页。

输入：
- 当前 app 已有功能
- 已确认限制：只做交互完善、功能闭环优先、考勤记录不要详情页

输出：
- 明确范围说明
- 明确“考勤记录 = 列表页闭环，不做详情页”

验收标准：
- 后续任务不再包含考勤详情页建设
- 不引入新业务模块

依赖关系：
- 无

执行类型：
- HITL（已明确）

### 任务 2：梳理现有功能的用户主流程
目标：
- 整理当前已有功能的使用路径，并同步去掉“考勤详情页”路径。

输入：
- 现有模块：登录、首页打卡、考勤列表、SOP、我的 / 设置

输出：
- 用户主流程清单

验收标准：
- 考勤流程不再包含“列表 → 详情”
- 每条流程都能说明入口、动作、反馈、结束状态

依赖关系：
- 依赖任务 1

执行类型：
- AFK

### 任务 3：识别各流程的交互缺口
目标：
- 找出当前页面交互上未闭环的地方，尤其是考勤列表页要承接详情页取消后的信息缺口。

输入：
- 任务 2 的主流程清单
- 当前页面现状

输出：
- 模块级交互缺口清单

验收标准：
- 每个缺口都落到具体页面和具体节点
- 不引入未确认业务规则

依赖关系：
- 依赖任务 2

执行类型：
- AFK

### 任务 4：定义每个模块的“闭环完成态”
目标：
- 明确本轮每个模块交互完善到什么程度算完成。

输入：
- 任务 3 的缺口清单

输出：
- 各模块完成态定义
- 其中考勤模块完成态应为：列表页即可完成最近记录查看，用户无需进入详情页也能理解主要信息，列表具备必要的空态、失败态、信息层级和返回一致性。

验收标准：
- 不再出现考勤详情页作为必需能力
- 每个模块的完成态都可验收

依赖关系：
- 依赖任务 3

执行类型：
- AFK

### 任务 5：拆分为独立可执行的交互完善任务包
目标：
- 把闭环目标拆成独立任务，便于后续逐项实施。

输入：
- 任务 4 的完成态定义

输出：
- 独立任务包，例如：登录页状态反馈完善、首页打卡状态流转完善、考勤列表页信息承载完善（替代详情页）、SOP 列表 / 详情状态回流完善、我的 / 设置页最小交互闭环完善。

验收标准：
- 每个任务包可以单独实施和验收
- 考勤任务包只围绕列表页，不含详情页

依赖关系：
- 依赖任务 4

执行类型：
- AFK

### 任务 6：按“功能闭环优先”排序实施顺序
目标：
- 先排出最影响可用性的交互闭环任务顺序。

输入：
- 任务 5 的任务包

输出：
- 排序后的实施顺序

验收标准：
- 先高频主流程，后低频辅助流程
- 考勤列表页完善优先于任何详情页构想

依赖关系：
- 依赖任务 5

执行类型：
- AFK

### 任务 7：确认第一阶段实施清单
目标：
- 确认第一阶段真正要落地的几个交互任务。

输入：
- 任务 6 的排序结果

输出：
- 第一阶段清单
- 暂缓项清单
- 验收口径

验收标准：
- 能明确先做哪几项
- 不再把考勤详情页纳入阶段目标

依赖关系：
- 依赖任务 6

执行类型：
- HITL

# 员工端请假流程-数据库与接口设计文档

## 1. 文档目标

本文档用于指导 `wmshr` 员工端请假流程的后端数据模型、接口、审批流与考勤联动设计。

本轮目标不是重做整套考勤系统，而是在当前项目已有移动端请假界面和考勤结算模型上，**补齐缺失的请假后端能力**。

设计原则：

- 请假申请与最终考勤结果分层保存：申请流单独落库，批准后才写入正式考勤结果。
- 后端负责统一审批状态、日期展开、跨月统计和考勤回写口径，前端只展示结果。
- 已批准请假必须进入现有 `attendance_records / attendance_calculation_results / monthly_attendance_summaries` 链路，不能做一套游离于考勤之外的“假数据历史”。
- 本轮优先满足当前员工端真实需求：申请、查看历史、查看本月请假汇总、后台审批。
- 遵循 YAGNI：本轮不设计多级审批、抄送、撤销重提、半天假/小时假等未确认能力。

关联现状依据：

- 移动端请假类型与返回结构：`apps/mobile/src/features/attendance/types.ts`
- 移动端请假接口入口：`apps/mobile/src/features/attendance/services/attendanceApi.ts`
- 移动端请假 UI：`apps/mobile/src/features/attendance/screens/AttendanceListScreen.tsx`
- 已有 mobile attendance 路由簇：`apps/admin/server/index.js`
- 已有考勤结果表与月汇总表迁移：`supabase/migrations/*attendance*`

## 2. 当前现状与缺口

### 2.1 已存在的前端契约

当前移动端已经直接调用以下接口：

- `GET /api/mobile/attendance/leave/summary`
- `GET /api/mobile/attendance/leave/history`
- `POST /api/mobile/attendance/leave/request`

对应前端类型：

```ts
export type LeaveType = 'personal' | 'sick' | 'annual' | 'special';
export type LeaveApprovalStatus = 'pending' | 'approved' | 'rejected';

export interface LeaveSummary {
  monthUsedDays: number;
  pendingCount: number;
}

export interface LeaveRecord {
  id: string;
  type: LeaveType;
  durationDays: number;
  startDate: string;
  endDate: string;
  reason: string;
  status: LeaveApprovalStatus;
}

export interface LeaveRequestPayload {
  type: LeaveType;
  startDate: string;
  endDate: string;
  reason: string;
}
```

### 2.2 已存在的后端能力

当前后端已落地并能工作的移动端考勤路由包括：

- `GET /api/mobile/attendance/today`
- `GET /api/mobile/attendance/records`
- `GET /api/mobile/home/summary`
- `POST /api/mobile/attendance/check-in`
- `POST /api/mobile/attendance/check-out`
- `POST /api/mobile/attendance/location`

同时数据库已存在并在生产逻辑中使用：

- `attendance_records`
- `attendance_calculation_results`
- `monthly_attendance_summaries`
- `employees`
- `employee_notifications`

### 2.3 当前缺口

截至本轮检查：

1. **前端已接好请假接口路径，但服务端未看到对应 leave 三个路由实现。**
2. **迁移目录未看到现成的 `leave_requests` 表。**
3. **现有考勤表支持 `leave / sick_leave` 状态，但没有“请假申请单”这一审批实体。**
4. **如果直接把请假历史塞进 `attendance_records`，会混淆“申请中”和“已批准”的边界。**

因此本轮设计必须新增一层 **请假申请主表**，并在批准后回写现有考勤结果链路。

## 3. 设计总览

## 3.1 分层模型

本轮采用两层数据模型：

### 第一层：申请层

新增 `leave_requests` 表，保存：

- 谁提交的请假
- 请什么类型的假
- 起止日期
- 原因
- 当前审批状态
- 审批备注

这层只代表“申请单”。

### 第二层：结果层

批准后把请假日期展开并同步到现有考勤结果链路：

- `attendance_records`
- `attendance_calculation_results`
- `monthly_attendance_summaries`

这层代表“正式生效的考勤结果”。

### 3.2 为什么不直接只用 attendance_records

如果只用 `attendance_records`：

- 无法表达 `pending / rejected`
- 无法保留审批人、审批备注、提交时间
- 无法区分“员工申请了”与“后台已经批准并计入考勤”

所以 `attendance_records` 只能承载**批准后的结果**，不能替代申请单。

### 3.3 为什么本轮先不拆 approval_logs

理论上可以再建一张 `leave_request_logs` 或 `leave_approval_logs` 记录状态变化历史。

但按当前需求：

- 员工端只要看申请状态与历史
- 后台只需要审批操作
- 目前没有二次流转、多级审批、撤回再审等复杂要求

因此本轮建议：

- **主表先直接保存当前状态与最后审批信息**
- 后续若要补完整审计，再增 `leave_request_logs`

这样能更快补齐当前真实缺口。

## 4. 数据模型设计

## 4.1 LeaveRequest 请假申请主表

### 表定位

保存员工端提交的请假申请，是请假流程唯一主实体。

### 建议表名

```sql
public.leave_requests
```

### 建议字段

```ts
interface LeaveRequest {
  id: string | number;
  ownerUserId: string;              // 租户/管理员账号
  employeeId: number;               // 申请员工
  type: 'personal' | 'sick' | 'annual' | 'special';
  startDate: string;                // YYYY-MM-DD
  endDate: string;                  // YYYY-MM-DD
  durationDays: number;             // 含首尾日
  reason: string;
  status: 'pending' | 'approved' | 'rejected';

  submittedAt: string;
  approvedAt: string | null;
  rejectedAt: string | null;

  approverUserId: string | null;    // auth.users.id
  approvalNote: string | null;

  createdAt: string;
  updatedAt: string;
}
```

### 字段说明

#### `owner_user_id`

- 类型：`uuid`
- 作用：与现有员工/考勤/薪资表保持同一租户隔离口径
- 约束：`not null references auth.users(id)`

#### `employee_id`

- 类型：`bigint`
- 作用：关联申请员工
- 约束：`not null references public.employees(id)`

#### `type`

- 类型：`text`
- 枚举值：`personal | sick | annual | special`
- 说明：必须和移动端 `LeaveType` 保持一致

#### `start_date / end_date`

- 类型：`date`
- 约束：`end_date >= start_date`
- 说明：移动端当前是整天请假，不支持时段

#### `duration_days`

- 类型：`integer`
- 说明：后端写入时统一计算，不信任前端
- 约束：`duration_days > 0`

#### `reason`

- 类型：`text`
- 约束：去掉首尾空格后不能为空

#### `status`

- 类型：`text`
- 枚举值：`pending | approved | rejected`
- 默认值：`pending`

#### `submitted_at`

- 类型：`timestamptz`
- 说明：业务上的提交时间

#### `approved_at / rejected_at`

- 类型：`timestamptz | null`
- 说明：只会二选一写入

#### `approver_user_id`

- 类型：`uuid | null`
- 说明：当前审批人是后台账号，不是员工账号，所以直接挂 `auth.users`

#### `approval_note`

- 类型：`text | null`
- 说明：用于批准备注或驳回原因

### 唯一/索引建议

```sql
create index if not exists idx_leave_requests_owner_employee_created
  on public.leave_requests(owner_user_id, employee_id, created_at desc);

create index if not exists idx_leave_requests_owner_status_created
  on public.leave_requests(owner_user_id, status, created_at desc);

create index if not exists idx_leave_requests_owner_employee_date_range
  on public.leave_requests(owner_user_id, employee_id, start_date, end_date);
```

### 核心约束建议

```sql
check (type in ('personal', 'sick', 'annual', 'special'))
check (status in ('pending', 'approved', 'rejected'))
check (end_date >= start_date)
check (duration_days > 0)
```

## 4.2 AttendanceRecord 的补充字段

为了让“已批准请假”能和现有考勤结果链路建立强关联，建议给 `attendance_records` 增一个可选字段：

```sql
alter table public.attendance_records
  add column if not exists leave_request_id bigint references public.leave_requests(id) on delete set null;
```

### 为什么需要这个字段

如果批准后只是写 `type = leave`：

- 后续无法知道这条请假考勤来自哪张申请单
- 无法在撤销/重审时精准回滚对应日期
- 无法避免重复批准时重复插入

### 字段作用

- 请假审批通过后，按日期展开写入的 `attendance_records` 全部回挂到同一个 `leave_request_id`
- 后续重批/驳回/删除时可以按申请单反查并清理

---

## 4.3 本轮不新增 LeaveSummary 表

移动端请假汇总目前只需要：

- `monthUsedDays`
- `pendingCount`

这两个值完全可以通过 `leave_requests` 聚合得到：

- `monthUsedDays`：统计当月已批准请假天数
- `pendingCount`：统计当前员工待审批申请数

所以本轮**不需要单独建请假汇总表**。

## 5. 审批通过后的考勤联动设计

## 5.1 核心原则

- `pending` / `rejected` 状态只留在 `leave_requests`
- 只有 `approved` 才进入正式考勤结果链路

## 5.2 批准后的写入方式

当申请被批准时：

1. 将请假日期区间展开为逐日列表
2. 针对每个日期 upsert `attendance_records`
3. 类型映射：
   - `sick` -> `sick_leave`
   - 其他请假类型 -> `leave`
4. `source` 建议写 `mobile`
5. `note` 建议写入固定说明，例如：`员工端请假审批通过` 
6. 同步写入 `leave_request_id`
7. 对受影响日期重算 `attendance_calculation_results`
8. 对受影响月份重算 `monthly_attendance_summaries`

### 建议映射

```ts
function mapLeaveTypeToAttendanceType(type: LeaveType): 'leave' | 'sick_leave' {
  return type === 'sick' ? 'sick_leave' : 'leave';
}
```

## 5.3 为什么批准后要写 attendance_records

因为当前系统已有考勤与薪资链路都依赖：

- `attendance_records`
- `attendance_calculation_results`
- `monthly_attendance_summaries`

若批准后只改 `leave_requests` 而不写考勤结果：

- 员工端打卡记录与月汇总会不一致
- 后台月度考勤汇总无法感知该假期
- 薪资端 `leave_count` 与有效出勤口径无法同步

## 5.4 重复申请与冲突规则

本轮建议规则：

### 不允许与已批准请假重叠

同一员工，如果新的申请日期与**已批准**请假区间重叠，直接拒绝提交。

### 与待审批申请重叠也拒绝

同一员工，如果新的申请日期与**待审批**申请重叠，也拒绝提交。

### 与已有人工考勤记录冲突的处理

如果某些日期已经存在：

- 正常打卡记录
- 人工调整记录
- 其他非请假考勤记录

则本轮建议：

- 员工端提交时直接拒绝
- 返回冲突日期列表
- 由后台人工先处理考勤冲突，再重新申请或直接后台改考勤

原因：当前项目考勤链路已经在用，静默覆盖正常打卡风险太高。

## 6. API 设计

## 6.1 员工端接口

### 6.1.1 获取请假汇总

```http
GET /api/mobile/attendance/leave/summary
Authorization: Bearer <employee-app-token>
```

#### 返回

```json
{
  "monthUsedDays": 3,
  "pendingCount": 1
}
```

#### 口径

- `monthUsedDays`：当前自然月内，**已批准**请假累计天数
- `pendingCount`：当前员工状态为 `pending` 的申请单数量

#### 备注

为了兼容现有前端，返回对象直接对齐 `LeaveSummary`，不包 `data` 外壳。

---

### 6.1.2 获取请假历史

```http
GET /api/mobile/attendance/leave/history
Authorization: Bearer <employee-app-token>
```

#### 返回

```json
[
  {
    "id": "123",
    "type": "sick",
    "durationDays": 2,
    "startDate": "2026-06-20",
    "endDate": "2026-06-21",
    "reason": "发烧就医",
    "status": "approved"
  }
]
```

#### 排序

- 按 `created_at desc`

#### 本轮范围

- 先直接返回数组，和当前前端保持一致
- 暂不改成分页结构，避免影响已接好的员工端 UI

后续若历史单量明显增大，再升级为分页接口。

---

### 6.1.3 提交请假申请

```http
POST /api/mobile/attendance/leave/request
Authorization: Bearer <employee-app-token>
Content-Type: application/json
```

#### 请求体

```json
{
  "type": "personal",
  "startDate": "2026-06-25",
  "endDate": "2026-06-26",
  "reason": "家中有事处理"
}
```

#### 成功返回

```json
{
  "id": "124",
  "type": "personal",
  "durationDays": 2,
  "startDate": "2026-06-25",
  "endDate": "2026-06-26",
  "reason": "家中有事处理",
  "status": "pending"
}
```

#### 后端校验

- `type` 必须合法
- `startDate/endDate` 必须为合法日期
- `endDate >= startDate`
- `reason.trim()` 不得为空
- 日期范围不得与该员工已有 `pending/approved` 请假重叠
- 日期范围不得与已有非请假考勤记录冲突

#### 行为

- 写入 `leave_requests`
- 默认 `status = pending`
- 不立即写入 `attendance_records`

---

## 6.2 后台接口

员工端想要真正可用，后台必须至少补三类接口。

### 6.2.1 请假申请列表

```http
GET /api/admin/leave-requests?status=pending&employeeId=12&page=1&pageSize=20
```

#### 返回建议

```ts
interface LeaveRequestListItem {
  id: number;
  employeeId: number;
  employeeName: string;
  employeeDept: string;
  type: 'personal' | 'sick' | 'annual' | 'special';
  durationDays: number;
  startDate: string;
  endDate: string;
  reason: string;
  status: 'pending' | 'approved' | 'rejected';
  submittedAt: string;
}
```

### 6.2.2 批准请假申请

```http
POST /api/admin/leave-requests/:id/approve
Content-Type: application/json
```

#### 请求体

```json
{
  "approvalNote": "同意，请安排交接"
}
```

#### 行为

- 校验当前状态必须是 `pending`
- 更新 `leave_requests.status = approved`
- 写入 `approver_user_id / approved_at / approval_note`
- 按日期展开写入 `attendance_records`
- 重算受影响 `attendance_calculation_results`
- 重算受影响 `monthly_attendance_summaries`

### 6.2.3 驳回请假申请

```http
POST /api/admin/leave-requests/:id/reject
Content-Type: application/json
```

#### 请求体

```json
{
  "approvalNote": "该日期已有正常出勤记录，请先联系管理员处理"
}
```

#### 行为

- 校验当前状态必须是 `pending`
- 更新 `status = rejected`
- 写入 `rejected_at / approver_user_id / approval_note`
- 不写 `attendance_records`

## 7. 服务层设计

## 7.1 移动端提交服务

建议新增：

```ts
submitLeaveRequest({ ownerUserId, employeeId, body })
```

职责：

- 参数校验
- 区间冲突校验
- 计算 `durationDays`
- 插入 `leave_requests`
- 返回移动端 `LeaveRecord`

## 7.2 员工端汇总服务

建议新增：

```ts
fetchEmployeeLeaveSummary(ownerUserId, employeeId)
fetchEmployeeLeaveHistory(ownerUserId, employeeId)
```

### `fetchEmployeeLeaveSummary`

聚合来源：`leave_requests`

```sql
- month_used_days = sum(duration_days) where status = 'approved' and date range overlaps current month
- pending_count = count(*) where status = 'pending'
```

说明：

`monthUsedDays` 不能只看 `start_date` 所在月，必须按自然月裁切请假区间。例如：

- 请假：`2026-06-30` ~ `2026-07-02`
- 6 月汇总：记 1 天
- 7 月汇总：记 2 天

因此服务层需要做“跨月区间裁切”。

## 7.3 后台审批服务

建议新增：

```ts
approveLeaveRequest({ ownerUserId, requestId, approverUserId, approvalNote })
rejectLeaveRequest({ ownerUserId, requestId, approverUserId, approvalNote })
```

其中 `approveLeaveRequest` 还应调用：

```ts
syncApprovedLeaveToAttendanceRecords(leaveRequest)
recalculateAttendanceForLeaveRequest(leaveRequest)
refreshMonthlySummariesForLeaveRequest(leaveRequest)
```

## 8. 与现有考勤链路的衔接

## 8.1 attendance_records 写入策略

建议策略：

- 每个请假日生成一条记录
- `type` 写 `leave / sick_leave`
- `in_time / out_time` 为空
- `source = 'mobile'`
- `note = 员工端请假审批通过`
- `leave_request_id = 当前申请单 ID`

## 8.2 attendance_calculation_results 生成策略

沿用现有日计算逻辑，不新增独立请假计算表。

即：

- 写入 `attendance_records` 后
- 使用现有考勤重算入口
- 让结果自然生成 `status = leave / sick_leave`

## 8.3 monthly_attendance_summaries 更新策略

沿用现有月汇总生成方式。

特别说明：

当前系统已经在月汇总中把：

- `leave`
- `sick_leave`

一并计入 `leave_count`。

这和员工端请假能力是兼容的，因此本轮不新增新的月汇总字段。

## 9. 数据库迁移建议

## 9.1 新增 leave_requests 表

建议新增迁移：

```sql
create table if not exists public.leave_requests (
  id bigserial primary key,
  owner_user_id uuid not null references auth.users(id) on delete cascade,
  employee_id bigint not null references public.employees(id) on delete cascade,
  type text not null check (type in ('personal', 'sick', 'annual', 'special')),
  start_date date not null,
  end_date date not null,
  duration_days integer not null check (duration_days > 0),
  reason text not null,
  status text not null default 'pending' check (status in ('pending', 'approved', 'rejected')),
  submitted_at timestamptz not null default now(),
  approved_at timestamptz,
  rejected_at timestamptz,
  approver_user_id uuid references auth.users(id) on delete set null,
  approval_note text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  check (end_date >= start_date)
);
```

## 9.2 给 attendance_records 增加 leave_request_id

```sql
alter table public.attendance_records
  add column if not exists leave_request_id bigint references public.leave_requests(id) on delete set null;
```

## 9.3 索引

```sql
create index if not exists idx_leave_requests_owner_employee_created
  on public.leave_requests(owner_user_id, employee_id, created_at desc);

create index if not exists idx_leave_requests_owner_status_created
  on public.leave_requests(owner_user_id, status, created_at desc);

create index if not exists idx_leave_requests_owner_employee_start_end
  on public.leave_requests(owner_user_id, employee_id, start_date, end_date);

create index if not exists idx_attendance_records_leave_request_id
  on public.attendance_records(leave_request_id);
```

### 第 1 步：先补数据库迁移

- 新建 `leave_requests`

### 第 2 步：补服务端 mobile 三个接口

- `leave/summary`
- `leave/history`
- `leave/request`

### 第 3 步：补后台审批接口

- 列表
- 批准
- 驳回

### 第 4 步：接入考勤回写与重算

- 审批通过时生成请假考勤
- 联动日结果和月汇总

### 第 5 步：最后补后台界面

- 审批列表
- 审批操作
- 冲突提示

## 11. 本轮明确不做

以下能力暂不纳入本轮数据库/API 设计：

- 多级审批
- 抄送/会签
- 取消申请 / 撤回申请
- 半天假 / 小时假
- 按余额扣减年假额度
- 附件上传（病历、证明）
- 通知推送到外部 IM
- 独立请假统计汇总表
- 完整状态流转审计日志表

这些都属于后续可以扩展的二阶段能力。

---

## 12. 最终结论

结合当前项目现状，最合理的方案是：

1. **新增 `leave_requests` 作为请假申请主表**
2. **保留现有 `attendance_records / attendance_calculation_results / monthly_attendance_summaries` 作为正式考勤结果链路**
3. **批准后才把请假写入考勤记录并触发重算**
4. **员工端继续沿用已接好的三个接口路径，不改前端契约**
5. **后台补审批接口和审批列表，而不是让员工端直写正式考勤结果**

这套设计能最大程度复用你当前项目已有的考勤与薪资基础设施，同时补齐员工端请假流程缺失的数据库和接口层。

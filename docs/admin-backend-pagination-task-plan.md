# Admin 列表改造任务安排

## 总体实施顺序
1. 统一分页契约
2. 员工管理改造
3. 考勤计算改造
4. 薪资核算改造
5. 统一验证

## 任务拆解
### 任务 1：统一分页契约
- 目标：明确前后端统一使用的分页返回结构
- 范围：`types.ts`、`lib/api.ts`、相关列表组件的分页状态
- 验收：三个列表都能消费统一或等价的分页响应结构
- 类型：AFK

### 任务 2：员工管理真实后端分页
- 目标：员工管理列表按后端分页加载
- 重点：
  - 接入 `fetchEmployeesPage`
  - 列表翻页触发重新请求
  - 搜索变化回第一页
  - 维持当前卡片展示语义
- 验收：员工列表不再本地 `slice` 整包员工数组作为主分页
- 类型：AFK

### 任务 3：考勤计算真实后端分页
- 目标：考勤计算列表按后端分页加载
- 重点：
  - 后端接口支持 `page/pageSize`
  - 前端筛选变化回第一页
  - 分页器使用后端总数
- 验收：考勤翻页重新请求接口，返回当前页窗口
- 类型：AFK

### 任务 4：薪资核算真实后端分页
- 目标：薪资核算列表按后端分页加载
- 重点：
  - 后端接口支持 `page/pageSize`
  - 月份/状态/关键字变化回第一页
  - 分页器使用后端总数
- 验收：薪资翻页重新请求接口，返回当前页窗口
- 类型：AFK

### 任务 5：统一验证
- 目标：确认三块改造都可构建通过
- 推荐验证：
  - `HOME=/Users/admin npm run build:admin`
  - 如有必要补充语法/类型检查
- 类型：AFK

## 依赖关系
- 任务 1 是 2/3/4 的前置
- 任务 2/3/4 完成后再做任务 5

## 实施原则
- 手术式改动，只改分页直接相关代码
- 保留现有业务规则、筛选口径和展示口径
- 每个模块改完立即回填进度文档
- 遇到阻塞时记录到 `PROJECT_ISSUES.md`

# 门户 / Admin / 登录页 国际化长文案兼容需求确认

## 用户需求
- 处理国际化后文案变长导致的截断问题。
- 本轮先覆盖：
  - 门户（portal / `apps/home`）
  - Admin（`apps/admin`）
  - 登录页（admin 登录页，`apps/admin` 内 `AuthScreen`）

## 当前确认口径
- 问题范围聚焦在 Web 端国际化后，英文或其他语言文案长度明显大于中文时引发的：
  - 直接截断
  - 溢出遮挡
  - 挤压按钮或导航
  - 布局在窄宽度下失衡
- 本轮目标是优先修正这三类页面里的高风险展示位，确保核心文案可读、可点击、布局不被打坏。

## 默认实施范围
1. 检查门户、Admin、登录页中高风险长文案区域。
2. 优先处理核心交互位：
   - 导航 / Tab / 侧栏入口
   - 主按钮 / 次按钮
   - 标题 / 描述文案
   - 登录页主操作区
3. 根据不同场景采用最小必要兼容策略，例如：
   - 允许换行
   - 去掉强制单行裁切
   - 允许容器最小高度或自适应高度
   - 仅在必要处补充长文案保护样式
4. 保持现有业务逻辑、接口和路由不变。
5. 用真实 lint / build 验证改动后工程仍可通过校验。

## 本次默认不做
- 不扩展到移动端
- 不改文案内容本身
- 不重做整套设计系统或全局样式规范
- 不新增一整套 tooltip / 展开交互，除非当前页面已具备且本次必须依赖
- 不处理用户未提及的非门户 / 非 admin / 非登录页区域

## 初始验收标准
- 门户、Admin、登录页中的核心长文案不再被无意义单行截断。
- 核心按钮、导航入口和标题在较长翻译下仍可读、可点击。
- 不出现明显遮挡、重叠、溢出或因换行导致的主布局崩坏。
- `npm run lint` 与 `npm run build` 通过。

## 执行状态
- 用户已明确要求：开始实施。

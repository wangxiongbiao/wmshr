# App 在线更新 - 任务安排

## 任务 1：定位现有 App 全局启动入口与版本读取方式
- 目标：确认版本检查应挂载在 App 的哪个全局层，以及本地版本号从哪里读取。
- 输入：当前移动端应用结构、Expo 配置、全局 Provider/导航入口。
- 输出：全局版本检查落点说明。
- 验收标准：明确版本检查的挂载层、执行时机、本地版本号来源。
- 依赖：无。
- 类型：AFK。

## 任务 2：设计并实现后端最新版本接口
- 目标：提供最新 Android 版本信息接口，字段仅包含 `version`、`content`、`url`。
- 输入：需求确认文档、后端现有 API 结构。
- 输出：一个可被 App 调用的最新版本接口。
- 验收标准：接口返回结构稳定、字段齐全、无额外未确认字段依赖。
- 依赖：任务 1（可并行理解，但实现前需明确 App 调用口径）。
- 类型：AFK。

## 任务 3：确定当前最小版本数据来源
- 目标：让后端接口有一个明确且最小可用的数据来源。
- 输入：后端现有配置/数据结构。
- 输出：版本号、版本内容、版本 URL 的实际来源方案。
- 验收标准：可以明确说明如何修改线上版本信息并让接口生效。
- 依赖：任务 2。
- 类型：HITL（若存在固定配置 / 数据库存储 / 后台配置多个实现口径，需要按最小改动选择）。

## 任务 4：实现 App 全局版本检查与可选择更新提醒
- 目标：App 在进入主流程前检查线上版本；版本不一致时弹出可选择更新的提醒。
- 输入：任务 1 的落点、任务 2 的接口、任务 3 的数据来源方案。
- 输出：全局更新提醒与可跳过逻辑。
- 验收标准：旧版本进入 App 时会看到更新提示，但允许用户选择暂不更新继续使用；版本一致时不受影响。
- 依赖：任务 1、任务 2、任务 3。
- 类型：AFK。

## 任务 5：实现下载与安装触发流程
- 目标：用户点击更新后，下载 APK 并自动拉起系统安装流程。
- 输入：任务 4 的更新提示 UI、APK 直链 URL。
- 输出：可从更新弹窗触发的下载/安装流程。
- 验收标准：点击更新后进入系统下载/安装链路；用户也可以选择暂不更新并继续使用 App。
- 依赖：任务 4。
- 类型：AFK。

## 任务 6：补最小异常处理与回归验证
- 目标：补齐接口失败、字段缺失、URL 无效、安装未完成等最小异常处理，并验证主流程。
- 输入：任务 2 至任务 5 的实现。
- 输出：通过验证的更新闭环与异常处理结果。
- 验收标准：至少覆盖同版本不弹、不一致提示更新、可跳过继续使用、接口失败、无效 URL。
- 依赖：任务 2、任务 4、任务 5。
- 类型：AFK。

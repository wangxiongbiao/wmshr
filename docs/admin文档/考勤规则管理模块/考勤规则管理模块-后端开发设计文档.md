# 考勤规则管理模块后端开发设计文档

## 1. 文档目标

本文档用于指导 `wmshr-admin` 考勤规则管理模块的后端数据模型、接口、校验规则和业务约束设计。

设计原则：

- 考勤规则是独立主数据，不再只依赖全局 `AppConfig`。
- 员工通过当前规则字段和规则历史引用考勤规则。
- 被员工引用的考勤规则允许直接编辑。
- 停用规则不强制员工更换规则。
- 本轮不设计迟到、早退、缺勤规则。

关联文档：

- `wmshr-admin-business-flow-data-relations.md`
- `wmshr-admin-field-gap-analysis.md`
- `admin文档/员工原理模块/员工管理模块-后端开发设计文档.md`
- `admin文档/考勤规则管理模块/考勤规则管理模块-前端开发设计文档.md`

## 2. 已确认业务规则

### 2.1 已被员工引用的规则允许直接编辑

- 后端不禁止编辑已被员工引用的考勤规则。
- 编辑后，关联员工后续考勤计算使用更新后的规则。
- 如果业务触发历史考勤重算，历史结果可能受规则变更影响。
- 后端可以返回关联员工数量，供前端展示风险提示。

### 2.2 支持查看关联员工

- 后端需要提供按考勤规则查询关联员工的接口。
- 接口只返回查看数据。
- 不通过该接口修改员工规则关系。

### 2.3 停用规则不强制员工更换规则

- `isActive = false` 表示不允许新关联。
- 已关联员工可以继续使用该规则。
- 历史考勤、历史薪资仍可追溯该规则。
- 后端停用规则时不要求检查并迁移员工。

### 2.4 生效结束时间允许为空

- `effectiveEndDate = null` 表示长期有效。
- 如果不为空，必须大于或等于 `effectiveStartDate`。

### 2.5 暂不设计迟到、早退、缺勤规则

本轮后端模型、接口、计算逻辑不包含：

- 迟到规则
- 早退规则
- 缺勤规则

## 3. 数据模型设计

## 3.1 AttendanceRule 考勤规则表

### 3.1.1 表定位

`AttendanceRule` 保存可被员工引用的考勤规则。规则用于员工日常考勤计算和加班计算。

### 3.1.2 字段定义

#### `id`

- 名称：考勤规则 ID
- 类型：`number` / `bigint` / `uuid`
- 是否必填：是
- 是否唯一：是
- 生成方式：后端生成
- 用途：主键，被员工当前规则、员工规则历史、考勤计算结果引用

#### `name`

- 名称：规则名称
- 类型：`string`
- 是否必填：是
- 校验：去除首尾空格后不能为空
- 建议唯一性：同一租户或同一业务范围内不重复
- 用途：列表展示、员工选择规则、历史记录展示

#### `isActive`

- 名称：是否启用
- 类型：`boolean`
- 是否必填：是
- 默认值：`true`
- 用途：控制是否允许新关联
- 注意：停用不影响已关联员工继续使用

#### `effectiveStartDate`

- 名称：生效开始日期
- 类型：`date`
- 是否必填：是
- 用途：判断规则自身有效期

#### `effectiveEndDate`

- 名称：生效结束日期
- 类型：`date | null`
- 是否必填：否
- 为空含义：长期有效
- 校验：不为空时必须大于或等于 `effectiveStartDate`

#### `startShift`

- 名称：上班时间
- 类型：`time` / `string HH:mm`
- 是否必填：是
- 示例：`08:30`
- 用途：标准上班时间

#### `endShift`

- 名称：下班时间
- 类型：`time` / `string HH:mm`
- 是否必填：是
- 示例：`17:30`
- 用途：标准下班时间

#### `breakStart`

- 名称：休息开始时间
- 类型：`time` / `string HH:mm`
- 是否必填：是
- 示例：`12:00`
- 用途：计算休息扣除

#### `breakEnd`

- 名称：休息结束时间
- 类型：`time` / `string HH:mm`
- 是否必填：是
- 示例：`13:00`
- 用途：计算休息扣除

#### `standardHours`

- 名称：标准工时
- 类型：`decimal`
- 是否必填：是
- 校验：必须大于 0
- 示例：`8`
- 用途：计算标准工时和加班时长

#### `overtimeEnabled`

- 名称：是否启用加班
- 类型：`boolean`
- 是否必填：是
- 默认值：`true`
- 用途：控制是否计算加班

#### `otHourlyFee`

- 名称：加班费标准
- 类型：`decimal`
- 是否必填：启用加班时必填
- 校验：大于或等于 0
- 示例：`50`
- 用途：计算加班费

#### `overtimeMinUnitHours`

- 名称：加班最小计量单位
- 类型：`decimal`
- 是否必填：是
- 固定值：`0.5`
- 用途：加班计薪时长按 0.5 小时为单位

#### `overtimeRounding`

- 名称：加班取整方式
- 类型：`enum`
- 是否必填：是
- 固定值：`floor_to_half_hour`
- 用途：加班计薪时长向下取整到 0.5 小时

#### `createdAt`

- 名称：创建时间
- 类型：`datetime`
- 后端维护

#### `updatedAt`

- 名称：更新时间
- 类型：`datetime`
- 后端维护

---

## 3.2 Employee 与 AttendanceRule 的关系

员工主表建议保留当前考勤规则字段：

```ts
attendanceRuleId: string | number;
```

含义：

- 表示员工当前使用的考勤规则。
- 新增员工或编辑员工规则时，只允许选择可新关联的规则。
- 停用规则不允许被新选择，但已关联员工可以继续保留该值。

---

## 3.3 EmployeeAttendanceRuleHistory 员工规则历史表

员工切换考勤规则时，应记录历史。

建议字段：

```ts
interface EmployeeAttendanceRuleHistory {
  id: string | number;
  employeeId: string | number;
  attendanceRuleId: string | number;
  effectiveStartDate: string;
  effectiveEndDate: string | null;
  createdAt: string;
  updatedAt: string;
}
```

规则：

- 同一员工同一时间只能有一条当前有效规则历史。
- 当前有效记录的 `effectiveEndDate` 为 `null`。
- 员工切换规则时，关闭上一条历史，再新增一条历史。
- 本表由员工管理模块维护，考勤规则管理模块只读取关联情况。

---

## 4. 不纳入本模块的数据

### 4.1 taxRate

- 税率属于薪资或扣款规则。
- 不纳入考勤规则表。

### 4.2 overtimeMultiplier

- 当前业务按固定小时加班费计算。
- 不按倍率计算。
- 不纳入考勤规则表核心字段。

### 4.3 dailyBreakMinutes

- 休息时长可由 `breakStart` 和 `breakEnd` 计算。
- 不作为独立存储字段。

### 4.4 currency

- 当前不作为考勤规则核心字段。
- 员工薪资币种和薪资结果币种由薪资模块统一处理。

## 5. 接口设计

## 5.1 获取考勤规则列表

### 请求

```http
GET /api/admin/attendance-rules
```

### 查询参数

- `keyword`：可选，按规则名称搜索
- `isActive`：可选，按启用状态过滤
- `page`：可选
- `pageSize`：可选

### 响应字段

```ts
interface AttendanceRuleListItem {
  id: string | number;
  name: string;
  isActive: boolean;
  effectiveStartDate: string;
  effectiveEndDate: string | null;
  startShift: string;
  endShift: string;
  breakStart: string;
  breakEnd: string;
  standardHours: number;
  overtimeEnabled: boolean;
  otHourlyFee: number;
  overtimeMinUnitHours: 0.5;
  overtimeRounding: 'floor_to_half_hour';
  relatedEmployeeCount: number;
  createdAt: string;
  updatedAt: string;
}
```

---

## 5.2 创建考勤规则

### 请求

```http
POST /api/admin/attendance-rules
```

### 请求体

```ts
interface CreateAttendanceRuleRequest {
  name: string;
  isActive: boolean;
  effectiveStartDate: string;
  effectiveEndDate: string | null;
  startShift: string;
  endShift: string;
  breakStart: string;
  breakEnd: string;
  standardHours: number;
  overtimeEnabled: boolean;
  otHourlyFee: number;
}
```

### 后端默认补充

```ts
overtimeMinUnitHours = 0.5;
overtimeRounding = 'floor_to_half_hour';
```

---

## 5.3 获取考勤规则详情

### 请求

```http
GET /api/admin/attendance-rules/:id
```

### 响应

返回完整 `AttendanceRule`，并建议附带：

```ts
relatedEmployeeCount: number;
```

---

## 5.4 更新考勤规则

### 请求

```http
PUT /api/admin/attendance-rules/:id
```

### 请求体

同创建请求。

### 业务规则

- 允许更新已被员工引用的规则。
- 不因为 `relatedEmployeeCount > 0` 拒绝更新。
- 后端返回更新后的规则详情。

---

## 5.5 启用考勤规则

### 请求

```http
PATCH /api/admin/attendance-rules/:id/enable
```

### 处理

- 设置 `isActive = true`。
- 更新 `updatedAt`。
- 返回更新后的规则。

---

## 5.6 停用考勤规则

### 请求

```http
PATCH /api/admin/attendance-rules/:id/disable
```

### 处理

- 设置 `isActive = false`。
- 不迁移员工。
- 不清空员工当前 `attendanceRuleId`。
- 不删除员工规则历史。
- 返回更新后的规则。

---

## 5.7 获取规则关联员工列表

### 请求

```http
GET /api/admin/attendance-rules/:id/related-employees
```

### 查询参数

- `currentOnly`：可选，是否只看当前有效关联，默认 `false`
- `page`：可选
- `pageSize`：可选

### 响应字段

```ts
interface AttendanceRuleRelatedEmployeeResponseItem {
  id: string | number;
  employeeNo: string;
  name: string;
  dept: string;
  role: string;
  status: 'active' | 'on_leave' | 'probation' | 'disabled' | 'resigned';
  relationStartDate: string;
  relationEndDate: string | null;
  isCurrentRelation: boolean;
}
```

### 查询来源

- 当前关联可从 `Employee.attendanceRuleId` 查询。
- 历史关联可从 `EmployeeAttendanceRuleHistory.attendanceRuleId` 查询。
- 如果同时需要当前和历史，后端需要去重或明确返回关系状态。

---

## 6. 校验规则

创建和更新时统一校验：

- `name` 去除首尾空格后不能为空。
- `effectiveStartDate` 必填。
- `effectiveEndDate` 可为空。
- `effectiveEndDate` 不为空时不能早于 `effectiveStartDate`。
- `startShift` 必填，格式为 `HH:mm`。
- `endShift` 必填，格式为 `HH:mm`。
- `breakStart` 必填，格式为 `HH:mm`。
- `breakEnd` 必填，格式为 `HH:mm`。
- `standardHours` 必须大于 0。
- `overtimeEnabled` 必须是布尔值。
- `overtimeEnabled = true` 时，`otHourlyFee` 必须大于或等于 0。
- `overtimeEnabled = false` 时，`otHourlyFee` 可保留但不参与计算。

---

## 7. 可新关联规则判定

员工新增或切换规则时，后端应只返回可新关联规则。

可新关联规则建议条件：

- `isActive = true`
- 当前日期不早于 `effectiveStartDate`
- `effectiveEndDate` 为空，或当前日期不晚于 `effectiveEndDate`

注意：

- 停用规则不允许新关联。
- 已关联员工继续使用停用规则，不受此条件影响。
- 历史记录展示不受 `isActive` 限制。

---

## 8. 考勤计算使用规则

考勤计算时不应只读取当前规则列表，而应按员工和考勤日期匹配规则。

推荐顺序：

1. 根据员工 ID 查询员工规则历史。
2. 找到考勤日期落入有效期的规则历史。
3. 根据 `attendanceRuleId` 查询考勤规则。
4. 检查考勤日期是否落入规则自身有效期。
5. 使用规则中的班次时间、休息时间、标准工时和加班规则计算。
6. 如果找不到规则，标记为考勤规则缺失异常。

加班计薪时长：

```text
加班原始时长 = 有效工时 - 标准工时
加班计薪时长 = Math.floor(加班原始时长 / 0.5) * 0.5
加班费 = 加班计薪时长 × otHourlyFee
```

如果 `overtimeEnabled = false`：

```text
加班计薪时长 = 0
加班费 = 0
```

## 9. 数据迁移建议

当前系统只有 `INITIAL_CONFIG`，没有独立规则表。

迁移时建议生成一条默认规则：

```ts
const DEFAULT_ATTENDANCE_RULE = {
  name: '默认白班规则',
  isActive: true,
  effectiveStartDate: '2026-01-01',
  effectiveEndDate: null,
  startShift: '08:30',
  endShift: '17:30',
  breakStart: '12:00',
  breakEnd: '13:00',
  standardHours: 8,
  overtimeEnabled: true,
  otHourlyFee: 50,
  overtimeMinUnitHours: 0.5,
  overtimeRounding: 'floor_to_half_hour',
};
```

已有员工如果没有 `attendanceRuleId`，可默认关联这条规则。

## 10. 错误码建议

- `ATTENDANCE_RULE_NOT_FOUND`：考勤规则不存在
- `ATTENDANCE_RULE_NAME_REQUIRED`：规则名称不能为空
- `ATTENDANCE_RULE_EFFECTIVE_START_REQUIRED`：生效开始日期必填
- `ATTENDANCE_RULE_EFFECTIVE_DATE_INVALID`：生效结束日期早于开始日期
- `ATTENDANCE_RULE_TIME_FORMAT_INVALID`：时间格式错误
- `ATTENDANCE_RULE_STANDARD_HOURS_INVALID`：标准工时错误
- `ATTENDANCE_RULE_OT_FEE_INVALID`：加班费标准错误

---

## 11. 验收标准

后端完成后需要满足：

- 可以创建考勤规则。
- 可以查询考勤规则列表。
- 可以查询考勤规则详情。
- 可以更新已被员工引用的考勤规则。
- 可以启用和停用考勤规则。
- 停用规则不强制员工迁移。
- 可以查询某条规则关联的员工。
- 生效结束时间允许为空。
- 不包含迟到、早退、缺勤规则字段。
- 不把税率、币种、加班倍率作为考勤规则核心字段。

# 考勤计算管理模块后端开发设计文档

## 1. 文档目标

本文档用于指导 `wmshr-admin` 考勤计算管理模块的后端数据模型、接口、计算服务和业务约束设计。

设计原则：

- 后端负责统一计算口径，前端只展示后端计算结果。
- 每日考勤必须按员工和考勤日期匹配当时有效考勤规则。
- 找不到有效规则或关键数据缺失时，生成异常结果，不静默计算。
- 月度考勤汇总是薪资核算模块的输入。
- 本模块不维护员工档案、不配置考勤规则、不计算最终工资。

## 2. 已确认业务规则

### 2.1 按考勤日期匹配规则

- 每条考勤记录必须按员工和考勤日期匹配当时有效的考勤规则。
- 不允许只读取员工当前规则直接计算历史考勤。
- 不允许只读取当前启用规则直接计算历史考勤。

### 2.2 找不到规则时标记异常

- 如果找不到员工当日有效规则，生成异常计算结果。
- 异常结果保留员工、日期、原始记录信息。
- 异常结果不进入薪资核算。

### 2.3 加班计薪时长

- 加班原始时长：`max(0, 有效工时 - 标准工时)`。
- 加班计薪时长按 `0.5 小时向下取整`。
- 如果考勤规则关闭加班，计薪加班时长为 0。

### 2.4 本轮不做迟到、早退、缺勤自动规则

- 不根据上班时间自动判定迟到。
- 不根据下班时间自动判定早退。
- 不做复杂缺勤规则。
- 迟到、早退、缺勤类型可以作为人工录入或人工调整状态保留。

### 2.5 考勤规则允许直接编辑

- 考勤规则被编辑后，历史重算可能改变历史结果。
- 后端重算接口必须按当前规则数据重新计算。
- 如果需要锁定历史快照，后续再单独设计，本轮不做规则快照。

## 3. 数据模型设计

## 3.1 AttendanceRecord 原始考勤记录表

### 表定位

保存员工原始打卡或人工录入记录。

### 字段定义

#### `id`

- 名称：考勤记录 ID
- 类型：`number` / `bigint` / `uuid`
- 是否必填：是
- 生成方式：后端生成

#### `employeeId`

- 名称：员工 ID
- 类型：`number` / `bigint` / `uuid`
- 是否必填：是
- 关联：`Employee.id`

#### `date`

- 名称：考勤日期
- 类型：`date`
- 是否必填：是
- 用途：匹配规则、参与月度汇总

#### `inTime`

- 名称：上班打卡时间
- 类型：`time | null`
- 是否必填：视考勤类型而定
- 说明：缺勤或假期可为空

#### `outTime`

- 名称：下班打卡时间
- 类型：`time | null`
- 是否必填：视考勤类型而定
- 说明：缺勤或假期可为空

#### `type`

- 名称：考勤类型
- 类型：`enum`
- 枚举值：`normal | late | early | absent | leave | sick_leave | overtime`
- 说明：本轮不自动判定迟到、早退、缺勤复杂规则

#### `source`

- 名称：考勤来源
- 类型：`enum`
- 枚举值：`manual | device | import | system`
- 默认值：`manual`

#### `note`

- 名称：备注
- 类型：`string`
- 是否必填：否

#### `createdAt`

- 名称：创建时间
- 后端维护

#### `updatedAt`

- 名称：更新时间
- 后端维护

---

## 3.2 AttendanceCalculationResult 每日考勤计算结果表

### 表定位

保存员工某一天的考勤计算结果。该结果是前端考勤计算列表的主要数据源，也是月度汇总的数据来源。

### 字段定义

```ts
interface AttendanceCalculationResult {
  id: string | number;
  employeeId: string | number;
  attendanceRecordId: string | number | null;
  date: string;
  attendanceRuleId: string | number | null;
  attendanceRuleName?: string;

  rawInTime: string | null;
  rawOutTime: string | null;
  rawHours: number;
  breakDeductionHours: number;
  validHours: number;
  standardHours: number;
  overtimeRawHours: number;
  overtimePayHours: number;

  status: 'normal' | 'leave' | 'absent' | 'manual_adjusted' | 'exception';
  hasException: boolean;
  exceptionReason: string | null;
  note: string;
  calculatedAt: string;
  createdAt: string;
  updatedAt: string;
}
```

### 唯一约束

建议：

```text
employeeId + date 唯一
```

同一员工同一天只保留一条当前计算结果。重新计算时更新该结果。

```ts
  canGeneratePayroll: boolean;
  blockedReason: string | null;

  createdAt: string;
  updatedAt: string;

```

### 唯一约束

建议：

```text
employeeId + yearMonth 唯一
```

## 4. 计算服务设计

## 4.1 单日计算输入

```ts
interface CalculateDailyAttendanceInput {
  employeeId: string | number;
  date: string;
}
```

## 4.2 单日计算流程

1. 查询员工。
2. 查询员工该日期的原始考勤记录。
3. 查询员工该日期有效的 `EmployeeAttendanceRuleHistory`。
4. 查询对应 `AttendanceRule`。
5. 检查考勤日期是否落入考勤规则自身有效期。
6. 如果规则缺失或无效，生成异常结果。
7. 如果原始记录缺失，生成缺勤结果。
8. 如果类型为假期或缺勤，工时为 0。
9. 计算原始工时。
10. 计算休息扣除时长。
11. 计算有效工时。
12. 计算标准工时。
13. 计算加班原始时长。
14. 计算加班计薪时长。
15. 保存或更新 `AttendanceCalculationResult`。
16. 返回计算结果。

## 4.3 原始工时计算

```text
rawHours = rawOutTime - rawInTime
```

跨天处理：

```text
如果 rawOutTime < rawInTime，则 rawOutTime 按次日处理，加 24 小时。
```

## 4.4 休息扣除计算

```text
breakDeductionHours = 实际打卡区间 与 规则休息区间 的重叠时长
```

注意：

- 当前前端代码已经按重叠区间扣除。
- 后端应复用该口径，并返回 `breakDeductionHours` 供前端核对。

## 4.5 有效工时计算

```text
validHours = max(0, rawHours - breakDeductionHours)
```

## 4.6 加班计算

```text
overtimeRawHours = max(0, validHours - standardHours)
```

如果规则启用加班：

```text
overtimePayHours = Math.floor(overtimeRawHours / 0.5) * 0.5
```

如果规则关闭加班：

```text
overtimePayHours = 0
```

## 4.7 异常结果生成

异常结果必须包含：

- `employeeId`
- `date`
- `attendanceRecordId`，如果有
- `hasException = true`
- `status = exception`
- `exceptionReason`
- 工时字段默认 0

异常原因示例：

- `EMPLOYEE_NOT_FOUND`
- `ATTENDANCE_RULE_NOT_FOUND`
- `ATTENDANCE_RULE_EXPIRED`
- `IN_TIME_MISSING`
- `OUT_TIME_MISSING`
- `TIME_FORMAT_INVALID`

## 5. 月度汇总服务设计

## 5.1 月度汇总输入

```ts
interface CalculateMonthlyAttendanceInput {
  employeeId?: string | number;
  yearMonth: string;
}
```

## 5.2 月度汇总流程

1. 确定自然月开始日期和结束日期。
2. 获取员工列表，或指定员工。
3. 确保目标日期范围内每日计算结果存在。
4. 汇总每日 `AttendanceCalculationResult`。
5. 计算总原始工时、总休息扣除、总有效工时、总标准工时。
6. 计算总加班原始时长、总加班计薪时长。
7. 统计异常记录数量、缺勤天数、假期天数、人工调整次数。
8. 判断是否可进入薪资核算。
9. 保存或更新 `MonthlyAttendanceSummary`。
10. 返回月度汇总结果。

## 5.3 是否可进入薪资核算

建议规则：

```text
canGeneratePayroll = exceptionCount === 0
```

如果不可进入薪资核算：

```text
blockedReason = '存在未处理异常考勤记录'
```

## 6. 接口设计

## 6.1 获取每日考勤计算结果列表

### 请求

```http
GET /api/admin/attendance-calculations
```

### 查询参数

- `yearMonth`：必填，格式 `YYYY-MM`
- `employeeId`：可选
- `status`：可选
- `hasException`：可选
- `page`：可选
- `pageSize`：可选

### 响应

返回分页后的 `AttendanceCalculationResult` 列表，并附带员工名称、员工编号和规则名称展示字段。

---

## 6.2 获取每日考勤计算结果详情

### 请求

```http
GET /api/admin/attendance-calculations/:id
```

### 响应

返回完整 `AttendanceCalculationResult`，并建议附带：

- 员工基础信息
- 原始考勤记录
- 使用的考勤规则核心参数

---

## 6.3 调整原始考勤记录

### 请求

```http
PUT /api/admin/attendance-records/:id
```

### 请求体

```ts
interface UpdateAttendanceRecordRequest {
  date: string;
  type: 'normal' | 'late' | 'early' | 'absent' | 'leave' | 'sick_leave' | 'overtime';
  inTime: string | null;
  outTime: string | null;
  note: string;
}
```

### 处理

- 更新原始考勤记录。
- 将 `source` 标记为 `manual`，或保留原来源并记录人工调整日志，具体实现可后续扩展。
- 触发该员工该日期重新计算。
- 更新对应月度汇总。
- 返回最新每日计算结果。

## 7. 校验规则

### 7.1 原始考勤记录校验

- `employeeId` 必须存在。
- `date` 必须为合法日期。
- `type` 必须为合法枚举。
- `inTime` 和 `outTime` 如果填写，必须为 `HH:mm`。
- `type = absent` 或 `type = leave` 时，允许 `inTime` 和 `outTime` 为空。
- `type` 非 `absent` / `leave` 时，缺少上下班时间应生成异常结果。

### 7.2 重算请求校验

- `yearMonth` 必须为 `YYYY-MM`。
- `date` 必须为 `YYYY-MM-DD`。
- `employeeId` 如果传入，必须存在。

## 8. 与其他模块关系

### 8.1 与员工管理模块

读取：

- 员工 ID
- 员工编号
- 员工姓名
- 员工状态
- 员工考勤规则历史

不修改员工档案。

### 8.2 与考勤规则管理模块

读取：

- 考勤规则 ID
- 规则名称
- 有效期
- 上班时间
- 下班时间
- 休息时间
- 标准工时
- 是否启用加班
- 加班费标准
- 加班最小计量单位
- 加班取整方式

不编辑考勤规则。

### 8.3 与薪资核算模块

输出：

- 月度总有效工时
- 月度总标准工时
- 月度总加班计薪时长
- 是否可进入薪资核算

薪资核算模块应读取 `MonthlyAttendanceSummary`，不应再直接遍历原始考勤记录计算工时。

## 9. 数据迁移建议

当前代码没有独立后端模型，只有前端内存数据。

迁移建议：

1. 将现有 `INITIAL_ATTENDANCE` 迁移为 `AttendanceRecord`。
2. 将 `empId` 映射为 `employeeId`。
3. 给历史数据补充 `source = manual`。
4. 基于默认考勤规则生成每日 `AttendanceCalculationResult`。
5. 基于每日结果生成 `MonthlyAttendanceSummary`。

---

## 10. 错误码建议

- `ATTENDANCE_RECORD_NOT_FOUND`：原始考勤记录不存在
- `ATTENDANCE_CALCULATION_NOT_FOUND`：考勤计算结果不存在
- `ATTENDANCE_SUMMARY_NOT_FOUND`：月度汇总不存在
- `EMPLOYEE_NOT_FOUND`：员工不存在
- `ATTENDANCE_RULE_NOT_FOUND`：找不到有效考勤规则
- `ATTENDANCE_RULE_EXPIRED`：考勤规则不在有效期内
- `IN_TIME_MISSING`：上班时间缺失
- `OUT_TIME_MISSING`：下班时间缺失
- `TIME_FORMAT_INVALID`：时间格式错误
- `YEARMONTH_INVALID`：年月格式错误

---

## 11. 验收标准

后端完成后需要满足：

- 可以保存和调整原始考勤记录。
- 可以计算单日员工考勤结果。
- 每日结果记录使用的考勤规则 ID。
- 每日结果包含原始工时、休息扣除、有效工时、标准工时。
- 每日结果包含加班原始时长和加班计薪时长。
- 加班计薪时长按 0.5 小时向下取整。
- 找不到有效规则时生成异常结果。
- 异常结果不进入薪资核算。
- 可以批量重新计算。
- 可以重新计算自然月考勤。
- 可以查询月度汇总。
- 月度汇总能作为薪资核算输入。
- 不在本模块维护员工档案、考勤规则或工资结果。

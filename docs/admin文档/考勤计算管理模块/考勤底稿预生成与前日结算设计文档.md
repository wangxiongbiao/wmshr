# 考勤底稿预生成与前日结算设计文档

## 1. 文档目标

本文档用于定义 `wmshr-admin` 考勤计算模块从“有打卡才生成数据”调整为“员工日历底稿预生成 + 前一日自动结算”的业务设计。

核心目标：

- 考勤计算列表不再依赖员工是否已经打卡才出现数据。
- 每个在职、已入职员工在业务日期内都应有一条可核对的每日考勤结果。
- 当天数据用于过程跟踪，前一天数据用于正式结算。
- 缺勤必须由系统自动发现，不能因为没有打卡记录而在列表中消失。
- 月度考勤汇总和薪资核算必须基于完整的每日考勤结果。

关联文档：

- `docs/wmshr-后台考勤底层彻底调整为v2逻辑方案.md`
- `docs/admin文档/考勤计算管理模块/考勤计算管理模块-后端开发设计文档.md`
- `docs/admin文档/考勤计算管理模块/考勤计算管理模块-前端开发设计文档.md`

## 2. 当前问题

当前考勤计算数据主要在以下场景生成：

- 员工端打卡。
- 后台手动新增考勤。
- 后台调整已有考勤。
- 保存考勤配置后重算。
- 薪资核算前触发月度重算。

这会导致一个核心问题：

```text
员工没有打卡
  ↓
没有 attendance_records 原始记录
  ↓
如果没有额外触发月度重算
  ↓
attendance_calculation_results 没有当天结果
  ↓
考勤计算列表看不到该员工缺勤
```

这不符合考勤管理目标。缺勤是最需要被系统主动暴露的异常，不能因为没有打卡记录而缺失。

---

## 3. 设计原则

### 3.1 以“员工 × 日期”为考勤基础

考勤计算结果应基于员工日历生成，而不是基于打卡记录生成。

```text
员工档案 + 入职日期 + 在职状态 + 日期
  ↓
每日考勤底稿 / 计算结果
```

打卡记录只是影响每日结果的输入之一，不应该决定是否存在每日结果。

### 3.2 当天是过程态，昨天及以前是结算态

- 当天考勤用于查看打卡进度。
- 前一天及更早日期用于正式结算。
- 系统应在业务地零点自动结算前一天考勤。

### 3.3 不批量预生成整月未来数据

只建议生成当天，最多可扩展为生成下一工作日。

不建议一次性生成整月未来日期，原因：

- 员工可能新入职、离职、禁用。
- 请假、病假、人工调整可能变化。
- 未来数据会造成大量无意义结果和后续回滚。
- 月度汇总和薪资应主要依赖已结算日期。

### 3.4 后端统一计算，前端只展示

前端不在浏览器内推导缺勤、餐补、服务费、工时、薪资。

前端只展示后端返回的：

- 每日考勤计算结果。
- 月度考勤汇总。
- 薪资核算结果。

## 4. 推荐状态模型

### 4.1 每日考勤计算状态

建议扩展或明确以下状态：

```ts
type AttendanceCalculationStatus =
  | "pending"
  | "checked_in"
  | "normal"
  | "late"
  | "early"
  | "absent"
  | "leave"
  | "sick_leave"
  | "overtime"
  | "manual_adjusted"
  | "exception";
```

### 4.2 状态含义

#### `pending`

当天底稿状态。表示员工当天应出勤，但尚未形成完整打卡结果。

适用场景：

- 当天零点预生成。
- 当天尚未上班打卡。

不应进入薪资结算。

#### `checked_in`

员工已上班打卡，但尚未下班打卡。

适用场景：

- 员工当天完成上班打卡后。

不应进入薪资结算。

#### `normal`

完整打卡且按规则计算为正常考勤。

可以进入月度汇总和薪资核算。

#### `absent`

结算时没有有效打卡记录，系统自动判定为缺勤。

可以进入月度汇总，但不产生有效工时、上班费用、餐补。

#### `leave` / `sick_leave`

后台手动录入或调整的假期/病假状态。

可以进入月度汇总，但是否计薪由后续薪资规则决定。当前规则下不产生餐补。

#### `manual_adjusted`

后台人工补卡或调整形成的结果。

是否进入薪资取决于调整后的有效工时和状态。

#### `exception`

数据不完整或规则无法正确计算时的异常结果。

不应静默进入薪资核算，需管理员处理。

## 5. 自动任务流程

### 5.1 执行时间

建议每天业务地零点后执行一次。

以泰国/缅甸业务时区为例：

```text
业务地 00:00 后执行
```

如果使用 Vercel Cron，应按 UTC 折算调度时间，并在服务端明确业务日期，避免本机 UTC 日期和业务地日期混淆。

### 5.2 任务顺序

```text
1. finalizeYesterdayAttendance()
2. prepareTodayAttendanceDrafts()
3. refreshMonthlyAttendanceSummary()
4. refreshPayrollIfNeeded()
```

### 5.3 结算前一天考勤

`finalizeYesterdayAttendance()` 负责正式结算前一天数据。

输入：

- 当前账号在职员工。
- 员工入职日期。
- 前一天日期。
- 前一天原始打卡记录 `attendance_records`。
- 账号级考勤配置 `attendance_config`。

处理规则：

```text
如果员工未入职或已离职/禁用：跳过
如果已有 leave / sick_leave / manual 记录：按人工记录重算
如果有完整上下班打卡：按规则计算 normal / late / early / overtime / exception
如果只有上班或只有下班打卡：生成 exception 或 absent，具体按业务规则配置
如果没有任何打卡记录：生成 absent
```

输出：

- upsert `attendance_calculation_results`
- 更新当月 `monthly_attendance_summaries`

### 5.4 生成当天考勤底稿

`prepareTodayAttendanceDrafts()` 负责生成当天过程态底稿。

输入：

- 当前账号在职员工。
- 员工入职日期。
- 今天日期。

处理规则：

```text
如果员工未入职或已离职/禁用：不生成
如果当天已有计算结果：不覆盖人工状态，只刷新必要展示字段
如果当天没有计算结果：生成 pending 底稿
```

底稿字段建议：

```ts
{
  attendance_record_id: null,
  date: today,
  raw_in_time: null,
  raw_out_time: null,
  raw_hours: 0,
  break_deduction_hours: 0,
  valid_hours: 0,
  standard_hours: 0,
  overtime_raw_hours: 0,
  overtime_pay_hours: 0,
  work_pay: 0,
  overtime_pay: 0,
  meal_allowance_amount: 0,
  total_pay: 0,
  status: "pending",
  has_exception: false,
  exception_reason: null,
  source: "system"
}
```

## 6. 打卡与人工调整流程

### 6.1 员工上班打卡

员工端上班打卡后：

```text
写入或更新 attendance_records
重新计算当天 attendance_calculation_results
刷新当天所在月份 monthly_attendance_summaries
```

如果当天已有 `pending` 底稿，应更新为：

```text
checked_in
```

或在现有计算函数无法表达 `checked_in` 时先标记为 `exception`，但推荐新增 `checked_in`，便于白天过程跟踪。

### 6.2 员工下班打卡

员工端下班打卡后：

```text
补全 attendance_records.out_time
重新计算当天 attendance_calculation_results
刷新当月 monthly_attendance_summaries
```

根据上下班时间和配置，状态变为：

- `normal`
- `late`
- `early`
- `overtime`
- `exception`

### 6.3 后台新增/调整考勤

后台手动新增或调整考勤后：

```text
upsert attendance_records
recalculateDailyAttendance(date)
recalculateMonthlyAttendance(yearMonth)
```

人工记录优先级高于系统底稿。

例如：

- 系统已生成 `pending`
- 管理员录入病假
- 当天结果应变为 `sick_leave`

## 7. 月度汇总规则

月度汇总应基于完整每日结果，但要区分过程态和结算态。

### 7.1 可进入汇总的状态

可统计：

- `normal`
- `late`
- `early`
- `overtime`
- `absent`
- `leave`
- `sick_leave`
- `manual_adjusted`
- `exception`

不建议作为最终薪资依据：

- `pending`
- `checked_in`

### 7.2 当月未结束时

当月汇总可以展示过程数据，但薪资核算时必须明确：

- 只核算已结算日期；或
- 允许核算全月，但 `pending` / `checked_in` 会阻塞生成正式工资。

推荐策略：

```text
薪资正式生成时，如果当月仍存在 pending / checked_in / exception，标记 blocked，不静默生成工资。
```

---

## 8. 薪资联动规则

薪资核算应依赖月度考勤汇总和每日计算结果。

### 8.1 自动夜间任务

如果已有夜间薪资任务，应在薪资计算前先执行：

```text
recalculateMonthlyAttendance(yearMonth)
```

但该函数需要升级为支持：

- 今天生成 `pending`
- 昨天及以前结算为最终状态
- 未来日期不生成

### 8.2 服务费与餐补

餐补规则：

```text
只有正常有效出勤才产生餐补。
absent / leave / sick_leave / pending / checked_in / exception 不产生餐补。
```

服务费规则：

```text
服务费仍在薪资核算阶段按员工档案服务费比例计算。
考勤底稿阶段不计算服务费。
```

## 9. 接口设计建议

### 9.1 新增或复用自动任务接口

建议新增：

```http
GET /api/cron/attendance-nightly
```

职责：

- 结算前一天考勤。
- 生成当天考勤底稿。
- 刷新受影响月份汇总。

需要 `CRON_SECRET` 授权。

### 9.2 后台手动触发接口

建议新增：

```http
POST /api/admin/attendance-calculations/run-nightly
```

请求体：

```json
{
  "date": "2026-06-06"
}
```

用途：

- 管理员手动补跑某一天。
- 排查线上任务未执行时立即恢复数据。
- 不暴露 `CRON_SECRET` 给前端。

### 9.3 月度重算接口调整

现有：

```http
POST /api/admin/attendance-calculations/recalculate-monthly
```

建议明确行为：

- 只处理业务日期今天及以前。
- 今天生成过程态。
- 昨天及以前生成结算态。
- 未来日期不生成，已有未来结果应删除或忽略。

## 10. 数据表调整建议

### 10.1 `attendance_calculation_results.status`

需要支持新增状态：

- `pending`
- `checked_in`

如果数据库使用 enum，需要新增 enum 值。

如果当前是 text 字段，需要同步前后端类型定义。

### 10.2 `source`

建议 `source` 支持：

- `manual`
- `mobile`
- `device`
- `import`
- `system`

`system` 用于系统自动生成的底稿和自动缺勤结算。

### 10.3 幂等约束

每日结果必须保证同一账号、同一员工、同一日期只有一条结果。

建议唯一约束：

```sql
unique(owner_user_id, employee_id, date)
```

自动任务和手动重算都必须使用 upsert，不允许重复插入。

## 11. 前端展示设计

### 11.1 考勤计算列表

新增状态展示：

- `pending`：待打卡
- `checked_in`：已上班，待下班

展示规则：

- `pending` 用中性色，不显示为缺勤。
- `checked_in` 用蓝色或信息色，提醒仍未完成。
- `absent` 用红色，表示已经结算为缺勤。

### 11.2 筛选

状态筛选应支持：

- 全部
- 待打卡
- 已上班待下班
- 正常
- 缺勤
- 假期
- 病假
- 异常
- 人工调整

### 11.3 操作按钮

建议保留或新增：

- `生成/刷新今日底稿`
- `结算昨日考勤`
- `重算当前月份`

其中自动任务失败时，管理员可手动补跑。

## 12. 边界规则

### 12.1 新员工入职

- 入职日前不生成底稿。
- 入职日当天开始生成 `pending`。

### 12.2 离职/禁用员工

- 离职/禁用后不再生成新底稿。
- 历史数据保留。
- 当前考勤列表默认不展示离职/禁用员工，除非后续明确需要历史筛选入口。

### 12.3 跨月零点

例如 7 月 1 日零点：

- 结算 6 月 30 日。
- 刷新 6 月月度汇总。
- 生成 7 月 1 日底稿。
- 刷新 7 月过程汇总。

### 12.4 自动任务漏跑

如果自动任务漏跑，管理员可以手动执行：

```text
POST /api/admin/attendance-calculations/run-nightly
```

补跑后应重新生成缺失的底稿和结算结果。

### 12.5 重复执行

自动任务必须幂等。

重复执行同一天任务时：

- 不重复插入行。
- 不覆盖人工调整、休假、病假等人工优先状态。
- 可刷新计算字段和汇总字段。

## 13. 推荐落地步骤

### 阶段 1：补齐状态和底稿生成能力

- 扩展前后端状态类型：`pending`、`checked_in`。
- 增加系统底稿生成函数：`prepareAttendanceDrafts(date, ownerUserId)`。
- 保证 `attendance_calculation_results` 按 `owner_user_id + employee_id + date` 幂等 upsert。

验收：

- 没有打卡记录时，也能看到当天在职员工 `pending` 行。

### 阶段 2：前一日结算

- 增加 `finalizeAttendanceDate(date, ownerUserId)`。
- 没有打卡记录的员工结算为 `absent`。
- 不完整打卡结算为 `exception` 或按业务规则转缺勤。

验收：

- 次日零点后，昨天未打卡员工自动出现在缺勤列表。

### 阶段 3：自动任务和手动补跑

- 新增 `/api/cron/attendance-nightly`。
- 新增后台手动触发接口。
- 配置 Vercel Cron。

验收：

- 自动任务可生成当天底稿并结算昨天。
- 手动接口可补跑指定日期。

### 阶段 4：前端展示

- 考勤列表支持 `pending` / `checked_in` 状态展示。
- 状态筛选补齐新状态。
- 增加手动补跑入口。

验收：

- 管理员能清楚区分“今天待打卡”和“昨天已缺勤”。

### 阶段 5：薪资联动验证

- 薪资生成前确保月度考勤汇总已刷新。
- 如果存在未结算状态，工资结果应 blocked。
- 餐补只对有效正常出勤生效。

验收：

- 未打卡不会漏薪资前置数据。
- 未结算考勤不会静默生成正式工资。

---

## 14. 成功判定

完成后应满足：

- 当天打开考勤页，所有在职且已入职员工都有考勤底稿。
- 次日零点后，前一天没有打卡的员工自动显示为缺勤。
- 员工打卡会更新已有底稿，而不是依赖打卡创建考勤计算行。
- 后台休假、病假、补卡能覆盖系统底稿。
- 月度汇总不漏缺勤天数。
- 薪资核算不会因为缺少考勤计算行而漏算或误算。
- 自动任务可重复执行且不会重复插入数据。

---

## 15. 一句话结论

考勤计算应从“打卡驱动生成”改为“员工日历驱动生成”：每天零点结算昨天、生成今天底稿；打卡和人工调整只更新底稿，不能决定考勤数据是否存在。

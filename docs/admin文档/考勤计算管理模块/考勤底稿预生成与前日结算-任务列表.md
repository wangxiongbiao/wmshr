# 考勤底稿预生成与前日结算任务列表

## 1. 文档目标

本文档用于承接 `考勤底稿预生成与前日结算设计文档.md` 的落地拆解，明确每个任务的目标、输入、输出、验收标准、依赖关系和执行方式。

本轮范围只覆盖：

- 考勤底稿预生成。
- 前日自动结算。
- 当天 `pending` 过程态展示。
- 历史缺勤自动暴露。
- 月度汇总与既有薪资联动核对。

本轮不包含：

- 国际化收尾。
- 其他后台模块重构。
- 薪资模块重写。
- 与考勤底稿无关的 UI 改版。

关联文档：

- `docs/admin文档/考勤计算管理模块/考勤底稿预生成与前日结算设计文档.md`
- `docs/admin文档/考勤计算管理模块/考勤底稿预生成与前日结算-进度管控.md`

---

## 2. 已确认业务规则

- 本轮范围：只做考勤底稿预生成与前日结算。
- 当天底稿初始状态：`pending`。
- 业务时区：泰国时间。
- 允许新增或调整数据库字段。
- 当天列表需要展示 `pending` 底稿。
- 优先级：人工调整 > 休假 / 病假 > 打卡记录 > 自动缺勤。
- 薪资联动已经做过，本轮只核对是否需要适配新底稿和结算流程。

---

## 3. 任务清单

### 任务 1：冻结本轮业务范围与关键规则

- 目标：确认本轮只做考勤底稿预生成与前日结算，避免扩大范围。
- 输入：
  - 当前业务确认结果。
  - 设计文档。
- 输出：
  - 本轮范围边界。
  - 状态命名确认。
  - 时区确认。
  - 优先级确认。
- 验收标准：
  - 不纳入国际化、其他后台模块重构。
  - 当天底稿状态确定为 `pending`。
  - 时区确定为泰国时间。
  - 优先级确定为：人工调整 > 休假 / 病假 > 打卡记录 > 自动缺勤。
- 依赖关系：无。
- 执行方式：HITL。

### 任务 2：梳理现有考勤相关数据模型

- 目标：确认哪些表需要新增字段、复用字段或调整逻辑。
- 输入：
  - 员工表。
  - 打卡原始记录表。
  - 考勤计算结果表。
  - 月度考勤汇总表。
  - 薪资核算相关表。
- 输出：
  - 字段影响清单。
  - 数据表职责边界。
  - 数据库迁移判断。
- 验收标准：
  - 明确 `pending` 状态存放位置。
  - 明确是否需要记录生成来源。
  - 明确是否需要记录结算时间。
  - 明确人工调整如何防止被覆盖。
- 依赖关系：任务 1。
- 执行方式：AFK 为主，字段取舍需要 HITL 确认。

### 任务 3：定义考勤状态与流转规则

- 目标：确定每日考勤从底稿到结算的完整状态流。
- 输入：
  - `pending` 状态确认。
  - 优先级规则。
  - 现有考勤状态。
- 输出：
  - 状态定义。
  - 状态流转规则。
  - 自动任务、打卡、后台调整分别如何改变状态。
- 验收标准：
  - 当天未打卡显示 `pending`，不是缺勤。
  - 前一天结算后未打卡显示缺勤。
  - 人工调整不会被自动任务覆盖。
  - 休假 / 病假优先于自动缺勤。
- 依赖关系：任务 2。
- 执行方式：AFK 出草案，HITL 确认。

### 任务 4：设计并实现底稿预生成服务

- 目标：为指定日期生成应出勤员工的考勤底稿。
- 输入：
  - 员工有效性规则。
  - 日期。
  - 状态流转规则。
- 输出：
  - 底稿生成服务。
  - 幂等处理逻辑。
  - 异常处理逻辑。
- 验收标准：
  - 当天每个应出勤员工都有一条 `pending` 记录。
  - 重复执行不会重复生成。
  - 不覆盖人工调整记录。
  - 离职 / 未入职 / 禁用员工不会被生成。
- 依赖关系：任务 2、任务 3。
- 执行方式：AFK。

### 任务 5：设计并实现前日结算服务

- 目标：每天按泰国时间结算前一天考勤。
- 输入：
  - 前一天底稿。
  - 打卡记录。
  - 休假 / 病假。
  - 人工调整。
  - 考勤规则。
- 输出：
  - 前日结算服务。
  - 缺勤生成逻辑。
  - 结算状态更新逻辑。
- 验收标准：
  - 有完整打卡时正常计算。
  - 无打卡时自动缺勤。
  - 休假 / 病假不会被判缺勤。
  - 人工调整不会被覆盖。
  - 重复结算结果一致。
- 依赖关系：任务 3、任务 4。
- 执行方式：AFK。

### 任务 6：调整员工端打卡更新流程

- 目标：员工打卡时更新当天 `pending` 底稿。
- 输入：
  - 当前打卡接口。
  - 当天底稿。
  - 状态流转规则。
- 输出：
  - 上班打卡状态更新。
  - 下班打卡状态更新。
  - 没有底稿时的处理策略。
- 验收标准：
  - 上班打卡后不再保持 `pending`。
  - 下班打卡后能完成工时与费用计算。
  - 没有底稿时不会静默失败。
  - 原始打卡记录仍保留。
- 依赖关系：任务 4、任务 5。
- 执行方式：AFK。

### 任务 7：调整后台人工新增 / 补卡 / 休假 / 病假流程

- 目标：让后台人工操作正确覆盖自动底稿和自动结算。
- 输入：
  - 当前后台考勤新增流程。
  - 当前后台考勤编辑流程。
  - 优先级规则。
- 输出：
  - 人工调整覆盖逻辑。
  - 休假 / 病假覆盖逻辑。
  - 修改后重算逻辑。
- 验收标准：
  - 人工调整优先级最高。
  - 手动休假 / 病假不会被自动任务覆盖。
  - 历史补卡能刷新对应日期结果。
  - 修改历史结果后能影响月度汇总。
- 依赖关系：任务 5。
- 执行方式：AFK 为主，关键业务行为 HITL 确认。

### 任务 8：调整后台考勤计算列表展示

- 目标：让管理员能看到当天待打卡和历史缺勤。
- 输入：
  - 当前考勤计算列表。
  - 新状态模型。
  - admin-v2 表格结构偏好。
- 输出：
  - 今日 `pending` 展示。
  - 历史缺勤展示。
  - 状态筛选调整。
  - 表格文案调整。
- 验收标准：
  - 今日未打卡员工显示待打卡 / 待完成。
  - 历史未打卡员工显示缺勤。
  - 不把当天早上员工误判为缺勤。
  - 表格结构不偏离现有 admin-v2 考勤明细样式。
- 依赖关系：任务 3、任务 4、任务 5。
- 执行方式：AFK，展示文案 HITL 确认。

### 任务 9：建立定时任务流程

- 目标：每天泰国时间零点后自动完成前日结算与当天底稿生成。
- 输入：
  - 底稿生成服务。
  - 前日结算服务。
  - 当前定时任务 / API 结构。
- 输出：
  - 每日自动任务。
  - 执行顺序。
  - 失败日志。
  - 重试策略。
- 验收标准：
  - 先结算前一天。
  - 再生成当天底稿。
  - 重复执行幂等。
  - 失败原因可追踪。
- 依赖关系：任务 4、任务 5。
- 执行方式：AFK。

### 任务 10：建立手动补跑能力

- 目标：允许管理员或运维补跑指定日期 / 月份。
- 输入：
  - 底稿生成服务。
  - 前日结算服务。
  - 月度汇总刷新逻辑。
- 输出：
  - 补跑指定日期能力。
  - 补跑指定月份能力。
  - 补跑结果反馈。
- 验收标准：
  - 可以补跑某一天。
  - 可以补跑某个月。
  - 补跑不会重复生成或重复计费。
  - 补跑后月度汇总正确刷新。
- 依赖关系：任务 5、任务 9。
- 执行方式：AFK，入口暴露方式 HITL 确认。

### 任务 11：核对月度汇总联动

- 目标：确认每日结果完整后，月度汇总仍然正确。
- 输入：
  - 每日考勤结果。
  - 当前月度汇总逻辑。
  - 当前餐补 / 工时 / 加班费用规则。
- 输出：
  - 月度汇总适配结论。
  - 需要调整的字段或逻辑。
- 验收标准：
  - 缺勤进入月度统计。
  - 休假 / 病假进入月度统计。
  - 标准工时、加班工时、餐补费用汇总正确。
  - `pending` 不被当作已结算数据计入薪资。
- 依赖关系：任务 5、任务 7。
- 执行方式：AFK。

### 任务 12：核对薪资联动是否需要调整

- 目标：检查已有薪资联动是否适配新考勤结算流程。
- 输入：
  - 当前薪资核算流程。
  - 当前“立即核算薪资”逻辑。
  - 月度汇总结果。
  - 新增 `pending` / 自动缺勤逻辑。
- 输出：
  - 薪资联动核对结论。
  - 是否需要调整的清单。
- 验收标准：
  - 薪资不读取当天未结算的 `pending`。
  - 薪资读取完整的历史考勤结果。
  - 前日结算后薪资相关数据不失真。
  - 历史补卡 / 人工调整后的薪资影响路径明确。
- 依赖关系：任务 11。
- 执行方式：AFK 检查，是否调整需要 HITL 确认。

### 任务 13：制定验证用例并执行验收

- 目标：用真实场景验证整套流程。
- 输入：
  - 新底稿逻辑。
  - 新结算逻辑。
  - 后台列表。
  - 月度汇总。
  - 薪资核算。
- 输出：
  - 验证用例清单。
  - 执行结果。
  - 问题清单。
- 验收标准：
  - 今日员工自动出现 `pending`。
  - 正常打卡能更新结果。
  - 未打卡员工次日变缺勤。
  - 休假 / 病假不被算缺勤。
  - 人工调整不会被覆盖。
  - 月度汇总正确。
  - 薪资联动无误。
  - 定时任务重复执行不产生重复数据。
- 依赖关系：任务 4 到任务 12。
- 执行方式：AFK 执行，最终 HITL 验收。

- `考勤底稿预生成与前日结算-进度管控.md` 中对应任务状态。
- 已完成内容。
- 实际输出物。
- 验证结果。
- 遗留问题。
- 下一步任务。

如果任务范围、验收标准或依赖关系发生变化，应先更新本文档，再继续执行后续任务。

# v3 三模块集成数据层替换实施计划

> 版本时间：2026-06-16 03:40:15
> 适用范围：`customers / goods / expenses`
> 前置状态：Phase 1（主系统骨架扩展）与 Phase 2（v3 UI 迁入当前 admin）已完成。

---

## 1. 计划目标

在保持当前已接入的 v3 UI 不大改版的前提下，把三个模块从“可展示的原型态页面”推进为“可真实读写的后台模块”：

1. `customers` 客户管理
2. `goods` 入库管理
3. `expenses` 费用管理 / 费用核销

目标结果：
- 去除 `localStorage` / mock 数据作为真实数据源
- 接入当前 `apps/admin` 的真实 API 层
- 接入 `apps/admin/server/index.js` 的真实后端处理
- 数据按当前 admin 的 `owner_user_id` 隔离
- 刷新后数据不丢失
- 不破坏已有 `dashboard / employees / attendance / payroll / sop`

## 3. 总体实施策略

### 3.1 基本原则
1. **不推倒重做 UI**：继续直接使用已接入的 v3 页面主体
2. **先真实化数据，再做细节美化**
3. **按模块顺序推进**：`customers → goods → expenses`
4. **优先复用当前 admin 的后端模式**：
   - `owner_user_id` 作用域隔离
   - `lib/api.ts` 前端请求封装
   - `server/index.js` 服务端聚合逻辑
5. **先收口字段和读写契约，再写实现**，避免页面字段和服务端字段反复来回改

### 3.2 推荐推进顺序
1. 先做 `customers`
2. 再做 `goods`
3. 最后做 `expenses`

### 3.3 原因
- `customers` 是 `goods` 的上游选择源之一
- `goods` 又会复用员工与客户数据
- `expenses` 的业务语义最复杂，放最后更稳妥

## 4. 分阶段任务拆解

## Phase 3：客户管理数据层替换

### 目标
把 `CustomerManager` 从本地原型态改成真实 API + 后端持久化。

### 需要触达的文件
- `apps/admin/src/components/CustomerManager.tsx`
- `apps/admin/src/App.tsx`
- `apps/admin/src/lib/api.ts`
- `apps/admin/src/types.ts`
- `apps/admin/server/index.js`
- 如需要：Supabase 对应表/读写逻辑关联代码

### 子任务
#### Task 3.1：梳理 Customer 页面当前字段模型
- 明确客户主表字段
- 明确店铺绑定字段
- 明确信用流水字段
- 区分：首期必做字段 / 可后置字段

#### Task 3.2：设计 customers API 契约
- `GET /api/admin/customers`
- `POST /api/admin/customers`
- `PUT /api/admin/customers/:id`
- 如需要：店铺、流水是嵌套写入还是拆子接口

#### Task 3.3：服务端接入 customers 持久化
- 在 `server/index.js` 增加 customers 读写逻辑
- 保证 `owner_user_id` 隔离
- 统一错误返回格式
- 处理首期空数据返回

#### Task 3.4：前端替换 CustomerManager 数据源
- 移除 `localStorage` 作为真实源
- 改为页面加载时请求 customers API
- 新增保存中、加载中、失败态
- 保存成功后刷新或乐观更新

#### Task 3.5：验证 customers 模块
- 新增客户
- 编辑客户
- 刷新页面后数据保留
- 不同用户隔离（如当前环境可验证）

### 阶段退出条件
- `customers` 页面不再依赖本地原型数据作为真实源
- 至少完成列表读取 + 新增/编辑闭环
- build / lint 通过
- 页面刷新后数据仍在

3. 客户信用流水首期可先按只读或随客户保存的简化结构处理

## 7. 验收标准

- 三模块都在当前 admin 中可访问

- build / lint / dev 验证通过

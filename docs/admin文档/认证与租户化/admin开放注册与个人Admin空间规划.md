# Admin 开放注册与个人 Admin 空间规划

> 目标：在保留现有 Google 登录能力的前提下，为 admin 端新增邮箱注册 / 邮箱登录，并让注册用户继续沿用现有“一个账号一套自己的企业数据”的后台模型，而不是重做一套全新的多租户系统。

## 1. 需求结论

当前需求的准确表达应当是：

1. 支持用户通过邮箱注册
2. 支持用户通过邮箱登录
3. 保留现有 Google 登录
4. 每个注册用户进入的是 **自己已有/新建的一套 admin 数据空间**
5. 不把所有注册用户导入同一个公共后台
6. 优先复用当前已存在的账号隔离与初始化链路

结合项目现状，这次工作更像是：

- 在现有 **Google 账号 = 一套自己的企业数据** 模型上
- 新增 **Email 认证入口**
- 并让 Email 用户也复用同样的 owner 级数据隔离与初始化流程

而不是先把系统整体重构成完整的 `workspace / membership / organization` 多租户平台。

---

## 2. 现状判断（基于当前代码）

当前 admin 架构已经具备明显的“单账号 owner 空间”特征，不是单一公共后台：

### 2.1 认证现状
- admin 前端当前通过 Supabase session 取登录态
- Google 登录是现有唯一成熟认证入口
- 前端 API 请求会带当前 session token

### 2.2 数据隔离现状
后端已有大量逻辑按 `owner_user_id` 隔离数据，例如：

- `workspace_bootstrap_states`
- `employees`
- `attendance_rules`
- `monthly_payroll_results`
- `employee_notifications`
- 以及多个 admin 业务接口写入时自动补 `owner_user_id`

这说明当前真实模型更接近：

- 一个登录账号
- 对应一个 owner
- owner 下面有一套自己的员工、规则、薪资、通知等数据

### 2.3 初始化现状
当前 admin 已有 workspace/bootstrap 概念：

- `POST /api/admin/workspace/bootstrap`
- `GET /api/admin/workspace/bootstrap-status`

说明系统已经存在“首次进入后台时初始化当前账号数据空间”的能力。

### 2.4 当前结论
因此，这次需求不应被定义为“从单租户后台升级为多租户后台”，更准确的说法是：

> **当前系统已经有“一账号一企业/一套后台数据”的雏形，这次主要要补的是邮箱认证入口，而不是重做数据隔离主干。**

---

## 3. 本次改造的正确范围

## 3.1 要做的事

1. 新增 Email 注册
2. 新增 Email 登录
3. 保留 Google 登录
4. 让 Email 注册用户也复用现有 owner 初始化流程
5. 让 Google / Email 最终都落到同一套 owner 数据隔离模型
6. 明确同邮箱下 Google 与 Email 的归并规则

## 3.2 不应默认展开的大改造
除非后续明确要做团队协作，否则本次不需要默认引入：

- `workspace_memberships`
- 多组织 / 多空间切换
- owner / admin / editor / viewer 多级角色体系
- 一个用户管理多个企业
- 成员邀请与协作权限矩阵

这些属于后续平台化能力，不是这次“补邮箱登录”的必选前置项。

---

## 4. 推荐架构定位

当前更适合把系统定位为：

## 4.1 单用户 Owner 模型
- 一个 auth user
- 一套以 `owner_user_id` 为归属键的后台数据
- 一套私有 admin 界面

## 4.2 本次新增能力
在这个模型上，把认证入口从：

- 仅 Google 登录

扩展成：

- Google 登录
- Email 注册
- Email 登录

## 4.3 第一版目标
**不改变“一个账号一套自己的企业数据”这个核心业务模型。**

也就是说，Email 用户注册后，不是进入新概念的共享组织系统，而是进入和当前 Google 用户一致的 owner 后台模式。

---

## 5. 推荐数据模型策略

> 第一版优先复用现有 `owner_user_id` 隔离体系，不强制把全库改成显式 workspace/tenant 结构。

### 5.1 继续复用现有业务归属模型
现有业务表如果已经通过 `owner_user_id` 隔离，就继续保持：

- 员工
- 考勤规则
- 薪资结果
- 通知
- 以及其他 owner 级数据

### 5.2 认证侧新增 Email 支持
需要补的主要是认证侧数据，而不是大规模业务表重构。

如果使用 Supabase 原生 email auth，建议优先沿用 Supabase 用户体系；如果项目内已有自定义认证辅助表，再按现有方式补充。

### 5.3 可接受的第一版做法
第一版只要满足下面三个条件即可：

1. Google 用户登录后仍进入自己原有的数据空间
2. Email 用户注册/登录后也进入自己的一套数据空间
3. 同一用户的不同登录方式不会无意中生成两套互相割裂的数据

---

## 6. 核心问题：Google 与 Email 的账号归并规则

这是本次最关键的设计点。

### 6.1 必须回答的问题
如果同一个邮箱：

- 曾经用 Google 登录过
- 后来又用 Email 注册/登录

系统要不要视为同一个 owner？

### 6.2 推荐规则
**推荐：同邮箱归并为同一个 owner。**

也就是说：

- `a@example.com` 用 Google 登录时创建过一套企业数据
- 之后 `a@example.com` 用邮箱密码登录
- 应继续进入同一套 owner 数据
- 不应再自动生成第二套企业数据

### 6.3 不推荐的规则
不建议让：

- 同一个邮箱的 Google 登录
- 同一个邮箱的 Email 登录

分别对应两套 owner 数据。

这会导致：

- 数据分叉
- 用户困惑
- 后续无法判断哪套才是“他自己的企业”

### 6.4 落地要求
因此在代码实现前，必须明确：

1. owner 的唯一标识是什么
2. Google 首登后 owner 与 auth user 的关系如何建立
3. Email 注册时如何查找是否已有同邮箱 owner
4. 如果已有同邮箱 Google 账号，Email 是否直接绑定到原 owner

---

## 7. 注册与登录流程规划

## 7.1 邮箱注册
前端输入：

- Email
- Password
- Confirm Password

建议流程：

1. 校验邮箱格式
2. 校验密码复杂度
3. 检查邮箱是否已存在
4. 创建 Email 认证账号
5. 建立 / 关联当前 owner 身份
6. 若该用户尚无 owner 数据空间，则复用现有 bootstrap 初始化
7. 登录成功后进入 admin

### 7.2 邮箱登录
建议流程：

1. 用户输入邮箱和密码
2. 认证成功后拿到 session
3. 查当前账号对应的 owner 身份
4. 若 owner 数据未初始化，则走 bootstrap
5. 进入 admin

### 7.3 Google 登录
Google 登录流程原则上保持不变，但需补一条一致性要求：

- Google 登录成功后，也要与 Email 登录共享同一套 owner 归属规则

不能让两个入口各自创建自己的数据归属逻辑。

---

## 8. 现有初始化链路的复用原则

当前项目已有：

- `workspace bootstrap`
- `bootstrap-status`

本次优先策略应当是：

### 8.1 复用而不是重写
- 不重写初始化业务数据逻辑
- 不另造一套“邮箱用户初始化”逻辑
- 尽量让 Email 用户登录后也调用同一套 bootstrap 检查/初始化流程

### 8.2 初始化触发条件
建议统一成：

- 当前账号首次进入 admin
- 当前 owner 尚未建立业务基础数据

则触发 bootstrap

### 8.3 目标
保证不论用户是：

- Google 首次登录
- Email 首次注册
- Email 首次登录

最终进入的都是同一套后台初始化机制。

## 9. 页面与交互改造范围

## 9.1 登录页
在保留现有 Google 登录入口的基础上，新增：

- Email 输入框
- Password 输入框
- Email 登录按钮
- 去注册链接

## 9.2 注册页
新增：

- Email
- Password
- Confirm Password
- 注册按钮

## 9.3 可后补但不是第一版必须项
- 忘记密码
- 重置密码
- 邮箱验证
- 登录风控

第一版可以先把注册 / 登录链路跑通。

## 10. 后端改造重点

本次后端重点不在业务表重构，而在认证与 owner 归属衔接。

### 10.1 重点模块
1. Email auth 接入
2. 登录后 owner 身份解析
3. 同邮箱归并规则实现
4. bootstrap 复用
5. session 与 admin 接口兼容

### 10.2 需要重点检查的风险
1. 当前 admin 是否隐式依赖 Google 特定字段
2. 当前 owner 是否直接等于 Supabase auth user id
3. 当前 bootstrap 是否只考虑了 Google 首登场景
4. 当前接口在 Email 登录后是否仍能稳定解析 `req.authUser.id`

---

## 11. 现在真正需要的改造级别

## 11.1 不是“重构成全新多租户系统”
这次不应默认按以下级别来做：

- 全库从 `owner_user_id` 改成 `workspace_id`
- 引入 membership 表
- 引入复杂租户角色体系
- 全面改写所有 admin 路由

## 11.2 而是“补认证入口 + 对齐 owner 归属规则”
第一版更合理的工作包是：

1. 增加 Email 注册/登录
2. 保持 Google 登录
3. 统一 owner 归属规则
4. 统一 bootstrap 触发逻辑
5. 确保同邮箱不会产生两套企业数据

---

## 12. 推荐实施阶段

## Phase 1：邮箱认证接入
目标：补齐 Email 注册 / 登录入口

包括：

1. 新增 Email 注册页
2. 新增 Email 登录表单
3. 接入 Email 认证
4. 保留 Google 登录

## Phase 2：账号归属统一
目标：让 Google / Email 都落到同一套 owner 模型

包括：

1. 明确 owner 唯一标识
2. 明确同邮箱归并规则
3. 补登录后 owner 解析逻辑
4. 验证 Google 与 Email 入口的一致性

## Phase 3：初始化与兼容验证
目标：确保新用户与老用户都能正常进入后台

包括：

1. Email 首次注册后 bootstrap
2. Google 老用户继续进入原有企业数据
3. 同邮箱双入口不会分裂数据
4. admin 业务接口对 Email session 正常工作

## Phase 4：安全增强（可后补）
包括：

1. 忘记密码
2. 邮箱验证
3. 登录限流
4. 审计日志
5. 2FA（如后续需要）

## 13. 第一版落地建议

如果目标是尽快上线，我建议第一版收敛为：

1. 保留现有 Google 登录
2. 新增 Email 注册
3. 新增 Email 登录
4. 继续复用 `owner_user_id` 数据隔离
5. 继续复用现有 bootstrap 初始化逻辑
6. 明确同邮箱归并为同一套 owner 数据
7. 暂不引入复杂团队/多空间模型

这样改动最小，也最贴近当前项目实际。

---

## 14. 当前规划结论

本需求的正确落点应当是：

- 不是“给固定 admin 补一个邮箱登录按钮”
- 也不是“先重做成完整多租户 SaaS 再做注册”
- 而是“在现有一账号一企业的 owner 架构上，补充 Email 注册 / 登录能力”

因此后续实施应按以下方向推进：

> **现有 owner 数据隔离主干尽量不动，重点补认证入口、账号归并规则和初始化复用。**

# 开发过程问题记录

## 2026-06-06：Admin 薪资核算语言切换后仍显示中文

### 问题现象

Admin 顶部语言切换控件可以切换语言，但进入“薪资核算 / 工资条”区域后，页面大部分文案仍显示中文。重启 Vite dev server 后现象仍存在，说明不是热更新缓存问题。

### 影响范围

- `apps/admin/src/components/PayrollTable.tsx`
- 主要影响薪资核算页、工资条弹窗、薪资状态、表头、按钮、空状态和部分导出/确认弹窗文案。

### 根因

`PayrollTable` 大量可见文案通过全局普通函数 `tAdmin(...)` 渲染。`tAdmin(...)` 直接读取 i18next 单例，能返回当前语言的文案，但它不是 React Hook，不会让组件订阅语言变化。

因此，Header 调用 `i18n.changeLanguage(...)` 后：

- 使用 `useTranslation(...)` 的 Header/Sidebar 等组件会自动重新渲染；
- 只调用 `tAdmin(...)`、但没有本地 `useTranslation(...)` 订阅的 `PayrollTable` 不一定重新渲染；
- 组件没有重新渲染时，屏幕上仍保留切换前语言的旧结果，看起来就像“语言切换无效”。

### 修复方式

在 `PayrollTable` 内显式加入：

```tsx
const { i18n } = useTranslation("admin");
const translationRenderLanguage = i18n.resolvedLanguage || i18n.language;
```

并把 `translationRenderLanguage` 挂到根节点 `data-i18n-language` 上。

这样做的目的不是改变布局，而是让 `PayrollTable` 订阅 react-i18next 的语言状态；语言变化时组件会重新渲染，渲染过程中所有 `tAdmin(...)` 调用会重新读取当前语言资源。

### 维护约束

- React 组件中的可见文案优先使用 `useTranslation("namespace")` 的 `t`。
- `tAdmin(...)` 只适合非 Hook 场景、回调、工具函数或逐步迁移期间的兼容层。
- 如果某个 React 组件继续大量使用 `tAdmin(...)`，必须在组件内显式订阅 `useTranslation(...)`，否则语言切换后可能不会刷新。
- 不要把 `tAdmin(...)` 放在模块顶层生成常量、模板数组或状态字典；这会在模块加载时缓存旧语言。

### 验证要求

修复这类问题后至少执行：

1. `npm --workspace @wmshr/admin run lint`
2. `npm run check:i18n`
3. 必要时执行 `npm --workspace @wmshr/admin run build`
4. 在浏览器中切换语言，观察目标组件是否随语言变化刷新。

## 2026-06-06：工资条带冒号字段切换语言后仍兜底中文

### 问题现象

工资条弹窗内“标准计薪时限（每日标准工时）:”“本月有效上班天数:”“本月累计上班工时:”“本月有效加班工时:”等字段，在翻译资源已经补齐后，切换到英文、泰文、印尼语、马来语、西班牙语或葡萄牙语时仍显示默认中文。

### 影响范围

- `apps/admin/src/lib/i18nText.ts`
- `apps/admin/src/components/PayrollTable.tsx` 工资条弹窗小时/天数字段
- 所有通过 `tAdmin(...)` 翻译、且 key 内包含英文冒号 `:` 的 Admin 文案

### 根因

`i18next` 默认把英文冒号 `:` 当作 namespace 分隔符。工资条这些文案使用中文 literal key，并且 key 末尾带 `:`，例如：

```tsx
tAdmin("本月有效上班天数:", { defaultValue: "本月有效上班天数:" })
```

虽然 `packages/i18n/src/namespaces/admin.ts` 已经存在对应翻译，但 `i18next.t(...)` 会先把这个 key 误解析为 namespace 形式，导致无法从 `admin` namespace 命中真实资源，最后返回 `defaultValue` 中文。

### 修复方式

在 `tAdmin(...)` 内固定：

```ts
return i18next.t(key, { ...options, ns: "admin", nsSeparator: false });
```

`tAdmin(...)` 本身已经明确固定 `ns: "admin"`，因此禁用 `nsSeparator` 不会影响 Admin 文案按 namespace 查找，反而能保护带冒号的工资条标签、表单标签和说明标签正常按完整 key 查询。

### 维护约束

- Admin 中文 literal key 允许包含 `:`，但必须通过禁用 namespace 分隔的翻译入口读取。
- 不要为了绕过这个问题删除业务展示中需要的冒号；应修复翻译解析配置。
- 如果未来新增其他固定 namespace 的 helper，也应评估是否需要 `nsSeparator: false`，避免 literal key 被误解析。
- 对带 `defaultValue` 的翻译调用，不能只看“界面有字”；必须验证切换语言后是否命中真实 locale 资源。

### 验证要求

修复后至少用探针验证：

```ts
tAdmin("本月有效上班天数:", { defaultValue: "本月有效上班天数:" })
```

在非中文语言下应返回对应语言翻译，而不是 `defaultValue` 中文。

## 2026-06-06：工资单出现“初始化示例”等开发文案

### 问题现象

工资条弹窗的一次性薪资项展示为“餐补 · 初始化示例”“绩效奖金 · 初始化示例”，员工姓名也存在“演示员工 A/B”这类种子数据文案。该问题不是 i18n 翻译缺失，而是后端初始化数据和历史数据库记录直接进入了工资单可见区域。

### 影响范围

- `apps/admin/server/index.js` 的 `bootstrapWorkspaceData(...)`
- `salary_adjustment_items.name/note`
- `attendance_records.note`
- `employees.name`
- `PayrollTable` 工资条中 `item.name` 与 `item.note` 的展示

### 根因

初始化流程最初把“初始化记录 / 初始化示例 / 演示员工”等开发阶段说明写进了业务表。工资单详情接口会原样返回 `salary_adjustment_items`，前端工资条又会拼接展示：

```tsx
{item.name}{item.note ? ` · ${item.note}` : ""}
```

因此这些开发文案会被当作正式业务备注显示在工资条里。

### 修复方式

- 初始化考勤备注改为业务描述：`正常加班`、`正常班次`、`缺下班卡`、`请假`、`标准工时`。
- 初始化薪资项备注改为空字符串，避免默认薪资项在工资条中展示无意义来源说明。
- 已清理当前数据库中的历史开发文案记录：
  - `salary_adjustment_items` 的 `初始化示例/初始化记录` 备注清空；
  - `attendance_records` 的初始化备注替换为业务备注；
  - `employees` 的 `演示员工 A/B`、`Demo Employee Updated` 替换为业务中性姓名。

### 维护约束

- 会进入业务页面、工资单、考勤明细、导出文件或客户可见接口的数据，不能写入“初始化、示例、演示、开发、demo、sample”等开发过程文案。
- 初始化数据如果必须存在，应使用业务中性名称和业务备注；开发来源只能放在状态表、日志或内部字段，不应进入可见业务字段。
- 修改初始化数据后，必须同时检查历史数据库记录，否则已生成工资条仍会继续显示旧文案。

### 验证要求

修复后至少检查以下表的可见字段：

- `salary_adjustment_items.name/note`
- `attendance_records.note`
- `employees.name/nickname/role/dept`
- `attendance_rules.name`

关键字扫描应不再命中：`初始化记录`、`初始化示例`、`示例`、`演示`、`开发`、`demo`、`sample`。

## 2026-06-06：考勤计算列表金额展示口径不统一

### 问题现象

考勤计算列表新增“服务费”列后，部分金额列只显示纯数字，合计列显示带币种/货币符号的金额，导致同一张表里的金额展示口径不统一，用户难以一眼判断不同币种员工的费用归属。

### 影响范围

- `apps/admin/src/components/AttendanceTable.tsx`
- 考勤计算列表可见金额列：
  - 上班费用
  - 餐补费用
  - 加班费用
  - 服务费
  - 合计费用

### 根因

考勤计算列表里同时存在两类金额格式化方式：

```tsx
formatMoneyNumber(...)
formatCurrency(...)
```

新增服务费列时，为了保持列表紧凑，非合计金额列沿用了纯数字格式；但列表已有合计费用会展示币种/货币符号，最终造成相邻金额列展示标准不一致。

### 修复方式

考勤计算列表的可见金额列统一使用：

```tsx
formatCurrency(金额, item.currency)
```

具体包括：

```tsx
formatCurrency(item.workPay, item.currency)
formatCurrency(item.mealAllowanceAmount, item.currency)
formatCurrency(item.overtimePay, item.currency)
formatCurrency(item.serviceFeeAmount, item.currency)
formatCurrency(getAttendanceTotalPayWithService(item), item.currency)
```

其中合计费用继续使用 `getAttendanceTotalPayWithService(...)`，确保合计包含服务费。

### 维护约束

- 考勤计算列表的可见金额列必须统一使用 `formatCurrency(金额, currency)`，不要在同一表格里混用纯数字金额和带币种金额。
- 合计费用必须通过 `getAttendanceTotalPayWithService(...)` 计算，避免后续新增服务费后又退回旧的 `totalPay` 展示。
- CSV 导出当前保留纯数字金额列，用于降低导出解析影响；如果后续要调整导出口径，应单独确认并同步说明。
- 修改金额展示时，应同时检查表格可见列、CSV 导出列和共享类型，避免列表与导出不一致。

### 验证要求

修复后至少执行：

1. `npm run lint`
2. `npm run build`
3. 前端访问检查：`http://127.0.0.1:3000/`
4. API 健康检查：`http://127.0.0.1:8788/api/health`

## 2026-06-26：Admin 登录页在手机浏览器看起来会“刷新”

### 问题现象

Admin 未登录页在手机浏览器中会出现整屏突然重排的现象，尤其在浏览器地址栏收起/展开、页面初次滚动或输入框聚焦拉起键盘时更明显，视觉上像是“页面刷新了一下”。

### 影响范围

- `apps/admin/src/components/AuthScreen.tsx`
- Admin 未登录页的全屏背景、卡片区和顶部操作条布局

### 根因

登录页外层容器和 `main` 同时使用了 `min-h-screen`，即基于 `100vh` 的高度模型。移动端浏览器会在地址栏、工具栏和软键盘状态变化时重新计算 `vh`，导致整个登录页高度被重新撑开/收缩。

这个现象不会触发真实的 HTTP 刷新，也不是邮箱表单默认提交造成的整页跳转：`AuthScreen` 的邮箱表单提交已经在 `handleEmailSubmit(...)` 里执行了 `event.preventDefault()`，登录按钮也是标准 `type="submit"`。因此当前问题属于移动端 viewport 单位导致的整页重排，而不是登录逻辑重新加载。

### 修复方式

登录页保留 `min-h-screen` 作为旧浏览器回退，同时在外层容器和 `main` 追加 `min-h-[100svh]`，让移动端优先使用稳定的小视口高度，降低地址栏/工具栏显隐时的整屏跳动。

继续复测后补充确认：登录表单输入框原本使用 `text-sm`（约 14px）。在 iPhone Safari 中，聚焦字体小于 16px 的输入框会触发浏览器自动缩放，视觉上同样会表现成“页面抖动/像刷新”。因此登录页邮箱、密码、确认密码输入框进一步调整为移动端 `text-base`、`sm` 及以上再回落为 `text-sm`，避免输入聚焦时的自动放大。

继续沿代码结构下钻后，发现移动端还有一层会持续放大抖动：登录页外层仍然是整屏布局，`main` 使用 `flex items-center` 强制垂直居中，根容器同时 `overflow-hidden`，并且小屏也继续渲染左侧高成本展示区。键盘弹出或地址栏高度变化时，浏览器为了保证输入框可见，会触发整页重新居中和平移，因此又补了一版“移动端稳定优先”修复：小屏改为顶部起排、允许纵向滚动，并隐藏左侧展示区，把大视觉布局只保留到 `lg` 及以上。

为彻底区分“真实页面刷新”和“仍在同一份单页应用上下文中，只是视口/焦点变化导致的抖动”，又在 `AuthScreen.tsx` 增加了只在 `import.meta.env.DEV` 下显示的登录页生命周期探针。探针会把 `sessionStorage` 持久化的 `loadCount`、Navigation Timing 的 `type`、`pageshow/pagehide/beforeunload`、`window.resize`、`visualViewport.resize/scroll`、`focusin` 以及 React mount/unmount 计数显示在页面左下角：如果只是抖动，`loadCount` 不会变化，最后事件会落在 `visualViewport.resize`/`focus:*`；只有整页重新执行 bundle 时，`loadCount` 才会增加。

在用户进一步判断“应该还是抖动”后，继续把移动端高成本视觉层做二分降级：小屏下隐藏两块 `blur-3xl` 背景光斑，顶部语言按钮/返回官网按钮/Logo 容器不再启用玻璃模糊，下拉语言菜单也只在 `lg` 及以上保留 `backdrop-blur`，登录主卡片则在移动端改为更稳定的深色实底 + `backdrop-blur-sm`，桌面端继续保持原有玻璃态。目的不是改 auth 逻辑，而是验证抖动是否主要来自移动端 blur/backdrop-blur 的高频重绘。

用户随后明确要求“关掉 StrictMode，并排查到底是哪里触发的问题”。因此继续沿入口层收缩：一方面在 `apps/admin/src/main.tsx` 临时移除 `React.StrictMode`，避免开发期双挂载把路由规范化和 auth 初始化都额外执行一轮，放大“像刷新”的观感；另一方面在 `App.tsx` 的 dev 分支增加 `sessionStorage` 诊断键，分别记录最近一次 `route-root-redirect / route-canonicalize / route-stable`、auth bootstrap 次数以及 `getSession / onAuthStateChange` 的最后结果，并让登录页左下角探针直接显示这些字段。这样手机上如果再次感觉“像刷新”，就能马上看到最后一次整屏切换是被路由 replace 触发，还是被 authLoading → 无会话登录页切换触发。

用户又补充“打开界面放着不动也会抖动或者刷新”。这条信息说明问题不再局限于输入聚焦或键盘弹出，因此继续排查 dev 层触发源，发现 `apps/admin/vite.config.ts` 默认开启 `hmr`（只在 `DISABLE_HMR=true` 时关闭），而配置注释本身就承认关闭 HMR/watch 可避免 flickering。于是把 `apps/admin/package.json` 的标准 `dev` 入口改成默认注入 `DISABLE_HMR=true`，同时新增显式 `dev:hmr` 入口，只在需要热更新时手动开启。这样手机端本地验收默认不再受 Vite HMR 重连/热替换干扰；若问题仍存在，就更能确认是页面自身链路而不是 dev server 层造成的“像刷新”。

用户随后要求“界面给我一个按钮，让我可以直接下载 F12 的输出日志”。为方便手机浏览器直接导出诊断数据，又在 `apps/admin/src/components/AuthScreen.tsx` 的 dev 探针里增加了前端日志采集与下载能力：拦截 `console.log/info/warn/error/debug`，同时捕获 `window error` 和 `unhandledrejection`，把最近一批日志存入 `sessionStorage`，并在探针面板上增加“下载日志”按钮。导出的 JSON 同时包含当前 URL、UA、boot 信息、probe 状态和 console 日志列表，这样即使手机端不方便打开 F12，也能一键把前端现场日志发回来继续排查。

用户上传首份手机端诊断日志后，首次拿到了决定性证据：`boot.navigationType` 为 `reload`，`loadCount=8`、`authBootstrapCount=8`，说明页面确实被重复整页重进；更关键的是日志里直接出现了 `[vite] server connection lost. Polling for restart...`。这意味着至少其中一类“像刷新”的来源不是登录页布局，而是本地 dev server / Vite 客户端连接中断后触发的页面重连。与此同时，日志里的 `routeNormalizeCount=0`，`appCause=auth-getSession-no-session`，只表明每次重进后 auth 初始化正常返回“无 session”，并没有证据表明路由规范化在循环触发。因此当前根因优先级已经收敛为：先排查并清理旧的 HMR dev 实例或其他导致 Vite 连接丢失的本地进程/重启行为，再看在稳定 dev server 下是否仍存在纯页面层抖动。

在替换掉旧的 3000 端口实例、确认新进程带有 `DISABLE_HMR=true` 后，用户又上传了第二份日志。新日志里 `routeNormalizeCount` 仍然是 `0`，`appCause` 仍是 `auth-getSession-no-session`，没有出现新的前端异常；但 `[vite] server connection lost. ## 2026-06-06：Supabase 登录态与一键发布全局 HOME 问题

### 问题现象

在生产一键发布过程中，首次按完整发布流程执行时，Supabase CLI 报错：

```text
Access token not provided. Supply an access token by running `supabase login` or setting the SUPABASE_ACCESS_TOKEN environment variable.
```

因此临时使用 `npm run deploy:prod -- --no-db` 完成前端生产发布，随后用户确认 Supabase 已登录后，再用全局 HOME 补跑完整发布。

- `scripts/deploy-production.sh`
- Supabase CLI 登录态读取
- 生产发布中的数据库迁移推送：`supabase db push --yes`

```text
/Users/admin/.hermes/profiles/wanglingling/home
```

```bash
HOME=/Users/admin <command>
```

```bash
HOME=/Users/admin supabase projects list
```

仍可能返回 access token 缺失，但在已绑定项目目录内执行以下命令可以成功：

```bash
HOME=/Users/admin supabase config push --yes
HOME=/Users/admin supabase db push --yes
HOME=/Users/admin supabase migration list --linked
```

因此不能只用 `projects list` 判断本项目的 Supabase 发布能力。

### 修复方式

确认 Supabase 已登录后，在项目根目录补跑完整发布，不再跳过 DB：

```bash
cd /Users/admin/Desktop/project/wmshr
HOME=/Users/admin npm run deploy:prod
```

本次完整发布已通过：

```text
Remote database is up to date.
```

并且本地与远端 migration 列表一致。

### 维护约束

- 涉及 GitHub、Vercel、Supabase 等外部 CLI 登录态时，优先显式使用 `HOME=/Users/admin`。

## 2026-06-16：考勤计算 / 薪资核算的离职筛选不能只改前端文案

### 问题现象

Admin 端“考勤计算”和“薪资核算”模块原有勾选项语义是“展示离职人员 / includeInactive”。本次需求要求统一改成和员工列表一致的“只看离职人员”。

如果只改复选框文案，不改请求参数链路，实际行为仍然只是把在职 + 离职一起放出来，无法真正收敛成“仅离职员工”。

### 影响范围

- `apps/admin/src/components/AttendanceTable.tsx`
- `apps/admin/src/components/PayrollTable.tsx`
- `apps/admin/src/lib/api.ts`
- `apps/admin/server/index.js`

### 根因

两条列表链路都没有现成的“员工状态筛选”参数：

- 考勤接口 `/api/admin/attendance-calculations` 里的 `status` 代表的是**考勤结果状态**，不是员工在职/离职状态；
- 薪资接口 `/api/admin/payroll-results` 只有 `includeInactive`，只能控制“是否把离职/停用员工混进结果”，不能表达“只看离职员工”；
- 前端员工远程搜索下拉原本也沿用 `includeInactive`，如果不一起调整，勾选“只看离职人员”后搜索候选仍会混入在职员工。

### 修复方式

本次统一新增并打通 `employeeStatus` 过滤语义：

- 前端 `AttendanceTable` / `PayrollTable` 改为使用 `resignedOnly` 本地状态；
- 勾选时请求统一传 `employeeStatus=resigned`，同时保留 `includeInactive=true` 以允许后端纳入离职员工；
- 后端在员工查询/筛选阶段单独按 `employee.status` 过滤，避免与考勤结果 `status` 混淆；
- 考勤页的“筛选员工”远程搜索跟随 `resignedOnly`，但“新增考勤记录”搜索仍保持默认在职员工范围，避免把录入入口误缩成只能给离职员工补记录；
- 薪资页员工搜索、分页缓存 key 和自动刷新依赖同步改为基于 `resignedOnly`。

### 维护约束

- 后续凡是业务上出现“只看离职 / 只看在职 / 只看 probation”这类员工身份筛选时，优先复用独立的 `employeeStatus` 语义，不要继续借用 `includeInactive`。
- `attendance-calculations` 的 `status` 仍然只表示考勤结果状态；不要再把它误当成员工状态参数。
- 涉及远程搜索、缓存 key、分页重置、空状态文案的筛选改造，必须前后端一起核对，不能只改 UI 文案。

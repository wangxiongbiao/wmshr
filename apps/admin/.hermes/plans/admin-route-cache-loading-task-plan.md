# Admin 模块路由缓存 + 加载体验优化任务安排

## 已确认目标
优化 admin 端一级模块切换时的数据加载体验：

1. 一级模块页面以独立路由承载。
2. 模块切换后不立即销毁旧页面/旧界面状态。
3. 返回访问过的模块时，优先显示该模块上一次成功渲染的数据界面。
4. 新数据在后台刷新，用户看到的是“旧数据 -> 新数据”的变化，而不是整页先变成 loading。
5. loading 主要表达“刷新中”，而不是“页面不可用”。

## 当前范围
先只覆盖 admin 一级模块：
- `dashboard`
- `employees`
- `attendance`
- `payroll`
- `sop`

## 当前不纳入本轮范围
1. 不处理登录页/Google OAuth 流程缓存。
2. 不处理弹窗详情页、二级子页面、深层子路由缓存。
3. 不强行一次性缓存所有局部分页/筛选/滚动细节，除非一级模块实现中已经天然保留。
4. 不顺手重构无关 API 层、视觉风格或文案系统。

## 建议实现思路
把本轮拆成两层：

### A. 页面承载层
- 每个一级模块变成独立页面组件/路由节点，而不是仅在 `App.tsx` 里条件渲染一大块内容。
- 引入模块级 keep-alive / route cache 容器，保证切换模块时旧页面实例不被立即卸载。

### B. 数据刷新层
- 每个模块保留“上一次成功数据”作为可见内容。
- 新请求触发时，不先清空页面主内容。
- loading 改为局部刷新状态：顶部提示、按钮 disabled、局部 skeleton、行内刷新标记等。
- 仅首次进入且无历史数据时，允许首屏 skeleton / loading。

## 任务拆分

### T1. 梳理一级模块路由与页面边界
- 状态：AFK
- 内容：
  - 核实现有 `/:lang/:tab` 如何映射到 dashboard / employees / attendance / payroll / sop。
  - 明确哪些模块已经是独立组件，哪些仍和 `App.tsx` 状态强绑定。
  - 设计一级模块页面壳层，确定后续 cache 容器挂载点。
- 验收：
  - 输出清晰的一级模块页面边界
  - 明确 keep-alive 容器应放在何处

### T2. 建立一级模块页面路由承载结构
- 状态：AFK
- 内容：
  - 将 5 个一级模块整理为明确的页面级承载单元。
  - 保持现有 `/:lang/:tab` 外部访问方式不变。
  - 保证语言切换与 tab 跳转逻辑不回退。
- 验收：
  - 5 个模块都有稳定页面承载结构
  - 路由切换不破坏当前语言/模块定位

### T3. 引入模块级路由缓存机制
- 状态：AFK
- 内容：
  - 在一级模块层实现 keep-alive / route cache。
  - 切换模块时保留已访问模块的页面实例，而不是立即卸载。
  - 明确缓存淘汰策略：本轮优先保留 5 个一级模块，不做复杂 LRU。
- 验收：
  - 已访问模块返回时无需重新走纯冷启动界面
  - 页面主区域不再因为切换而先变成空白 loading

### T4. 改造模块数据加载策略为“保留旧数据 + 后台刷新”
- 状态：AFK
- 内容：
  - 逐个梳理 dashboard / employees / attendance / payroll / sop 的数据加载入口。
  - 改为：已有成功数据时，刷新前不清空主内容。
  - 区分“首次加载”和“已有数据刷新中”两种状态。
- 验收：
  - 已访问模块再次进入时先显示旧数据
  - 新请求返回后界面平滑更新

### T5. 统一 loading 交互语义
- 状态：AFK
- 内容：
  - 将整页阻塞式“正在加载...”替换为更弱打断的刷新提示。
  - 页面已有内容时，优先保留内容，只对局部区域、操作按钮、顶部状态做 loading 表达。
  - 首次无数据时才保留必要 skeleton / loading。
- 验收：
  - loading 表现为“刷新中”而不是“页面消失”
  - 用户能持续感知当前所在模块和已有数据内容

### T6. 路由缓存与数据刷新的联动验证
- 状态：AFK
- 内容：
  - 验证 5 个一级模块之间的来回切换。
  - 验证语言切换后缓存/页面定位是否仍正确。
  - 验证刷新浏览器、前进后退、重新进入已访问模块时的表现。
- 验收：
  - 模块切换体验符合“先旧后新”目标
  - 路由与语言逻辑不回归
  - lint / build / 必要运行验证通过

## 推荐实施顺序
1. T1 页面边界梳理
2. T2 页面路由承载结构
3. T3 模块级路由缓存
4. T4 数据刷新策略改造
5. T5 loading 交互统一
6. T6 联动验证

## 依赖关系

- T3 依赖 T2
- T4 依赖 T2，部分依赖 T3
- T5 依赖 T4
- T6 依赖 T3 + T4 + T5

## 成功判定
1. admin 五个一级模块仍通过现有 `/:lang/:tab` 正常访问。
2. 已访问模块在切换回来时优先显示旧界面/旧数据。
3. 模块刷新时主界面不被整块 loading 替换。
4. 用户能看到数据刷新带来的变化过程。
5. 语言切换、前进后退、浏览器刷新不破坏模块定位与缓存策略。
6. lint / build / 必要运行验证通过。

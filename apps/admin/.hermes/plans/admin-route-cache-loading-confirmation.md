# Admin 模块路由化 + 路由缓存加载体验需求确认

## 目标
优化 admin 端各模块切换时的数据加载体验，避免每次切换页面都先看到一块生硬的 loading 占位。你希望切换后先保留并展示旧界面/旧数据，再在后台加载新数据，让用户能看到数据逐步变化，而不是页面突然清空。

## 项目路径
- `/Users/admin/Desktop/project/wmshr/apps/admin`

## 你当前表达出来的核心诉求
1. **每个模块一个独立路由**
   - 不再只把模块切换理解成单纯的本地 tab 状态。
   - 每个模块页面都应该有自己的页面路由承载。

2. **做路由缓存**
   - 模块切换后，不希望旧页面组件立刻被销毁。
   - 你更希望页面实例、已展示数据、滚动位置或至少主要界面状态能被保留。

3. **切换后先看到旧数据界面，再异步加载新数据**
   - 不是“点击模块 -> 整页空白/骨架/纯 loading -> 新数据出现”。
   - 而是“点击模块 -> 先看到该模块上一次的界面和数据 -> 同时触发刷新 -> 用户看到数据更新结果”。

4. **loading 应该变成‘数据刷新中的状态’，而不是‘页面不可用的状态’**
   - loading 需要尽量局部化、弱打断。
   - 用户仍能感知当前在哪个模块、当前已有哪批数据、哪些数据正在刷新。

## 当前实现（已核实）
1. admin 现在已经有 `/:lang/:tab` 这种一级路由。
2. 但当前模块切换后，是否保留旧页面实例/旧数据，并不是由专门的路由缓存层管理。
3. 多个模块各自管理自己的加载状态，例如：
   - `Dashboard.tsx` 自己维护 `loading`
   - `AttendanceTable.tsx` 自己维护 `loading`
   - `PayrollTable.tsx` 自己维护 `loading`
   - `SopManager.tsx` 自己维护 `isLoadingSops`
4. 一些模块在加载时仍会直接用“正在加载...”替换主体区域，因此交互观感偏硬。

## 问题本质
你现在提的不是单纯“换个 loading 样式”，而是一个更上层的交互模型调整：

1. **页面切换模型**要从“切 tab 即重建页面”转成“切路由但页面可缓存”。
2. **数据加载模型**要从“先清空再等新数据”转成“先显示上次内容，再后台刷新”。
3. **用户感知**要从“我在等页面出现”转成“我已经在这个页面里，只是数据正在更新”。

## 建议按两层拆开理解

### A. 路由层诉求
- 每个模块一个独立页面路由
- 切换模块时允许保留页面实例或页面缓存
- 后续可扩展页面级缓存、滚动恢复、返回恢复

### B. 数据层诉求
- 模块进入时默认优先展示上一次成功数据
- 同时触发后台刷新
- 加载态尽量用顶部进度、局部刷新标记、按钮 disabled、列表局部 skeleton 等方式体现
- 避免整页被“正在加载...”硬替换

## 你这次需求的隐含验收标准
如果按你的意思落地，最终至少应满足：

1. 从一个模块切到另一个模块时，页面不会先退化成一块空白 loading。
2. 回到访问过的模块时，优先看到该模块上一次的界面内容。
3. 模块数据刷新时，用户能看到“旧数据 -> 新数据”的变化过程。
4. loading 主要表现为“刷新中”，而不是“整页不可看”。
5. 路由仍然是页面状态入口，支持刷新、分享、前进后退。

## 建议结论（当前理解）
我对你需求的当前理解是：

- **admin 一级模块页面继续以路由为主入口**
- **每个一级模块都变成独立页面实例，支持页面缓存**

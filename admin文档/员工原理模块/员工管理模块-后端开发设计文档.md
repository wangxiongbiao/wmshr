# 员工管理模块后端开发设计文档

## 1. 文档目标

本文档用于指导 `wmshr-admin` 员工管理模块的后端接口、数据模型、校验规则和业务约束设计。

设计原则：

- 以当前前端已有员工字段为基础。
- 当前缺失但业务已确认的字段，从业务文档补齐。
- 员工管理作为基础主数据模块，服务考勤计算和薪资核算。
- 员工删除采用停用/软删除，不物理删除历史数据。

关联文档：

- `wmshr-admin-business-flow-data-relations.md`
- `wmshr-admin-field-gap-analysis.md`
- `admin文档/员工原理模块/员工管理模块-前端开发设计文档.md`

---

## 2. 已确认业务规则

### 2.1 员工编号

- 员工编号由后端自动生成。
- 员工编号必须唯一。
- 前端不允许手动指定或覆盖员工编号。

### 2.2 联系方式

- 联系方式只保留手机号。
- 不设计邮箱、紧急联系人等字段。

### 2.3 员工状态

员工状态采用：

- 在职：`active`
- 休假：`on_leave`
- 试用：`probation`
- 停用：`disabled`
- 离职：`resigned`

### 2.4 历史数据保留

- 离职员工允许继续出现在历史薪资中。
- 离职员工允许继续出现在历史考勤中。
- 停用员工允许继续出现在历史薪资中。
- 停用员工允许继续出现在历史考勤中。

### 2.5 删除规则

- 员工删除为停用/软删除。
- 不做物理删除。
- 停用/离职不影响历史考勤、历史薪资、规则历史追溯。

### 2.6 计薪方式

- 固定工资和时薪二选一。
- `salaryType = fixed` 时只使用固定工资。
- `salaryType = hourly` 时只使用时薪。

## 3. 数据模型设计

## 3.1 Employee 员工主表

### 3.1.1 表定位

`Employee` 是员工主数据表，保存员工基础信息、当前状态、当前考勤规则、计薪方式和薪资基础信息。

### 3.1.2 字段定义

#### `id`

- 名称：员工 ID
- 类型：`number` / `bigint`
- 是否必填：是
- 是否唯一：是
- 生成方式：后端生成
- 用途：员工主键，被考勤记录、薪资记录、规则历史引用

#### `employeeNo`

- 名称：员工编号
- 类型：`string`
- 是否必填：是
- 是否唯一：是
- 生成方式：后端自动生成
- 建议格式：`EMP000001`
- 用途：业务识别、搜索、工资单展示、接口对接
- 约束：创建员工时生成，不允许前端覆盖

#### `name`

- 名称：姓名
- 类型：`string`
- 是否必填：是
- 校验：去除首尾空格后不能为空
- 用途：员工展示、搜索、考勤/薪资展示

#### `gender`

- 名称：性别
- 类型：`enum`
- 是否必填：是
- 枚举值：
  - `male`
  - `female`

#### `country`

- 名称：国家/地区
- 类型：`enum`
- 是否必填：是
- 当前枚举值：
  - `MM`
  - `TH`
  - `CN`
  - `VN`
  - `KH`

#### `phone`

- 名称：手机号
- 类型：`string`
- 是否必填：是
- 校验：
  - 去除首尾空格
  - 暂不强制具体国家格式
  - 允许带 `+国家区号`

#### `role`

- 名称：岗位
- 类型：`string`
- 是否必填：是
- 当前候选：
  - 拣货员
  - 打包员
  - 叉车工
  - 质检员
  - 组长
  - 仓管员
- 后续可迁移为岗位字典

#### `dept`

- 名称：部门/区域
- 类型：`string`
- 是否必填：建议是
- 用途：员工归属、搜索、后续分组统计

#### `joinDate`

- 名称：入职日期
- 类型：`date`
- 是否必填：是
- 格式：`YYYY-MM-DD`
- 用途：员工档案、入职当月考勤/薪资起算

#### `status`

- 名称：员工状态
- 类型：`enum`
- 是否必填：是
- 默认值：`active`
- 枚举值：
  - `active`
  - `on_leave`
  - `probation`
  - `disabled`
  - `resigned`

状态含义：

- `active`：在职，参与新考勤和新薪资。
- `on_leave`：休假，考勤中可显示休假，薪资处理后续按薪资规则。
- `probation`：试用，参与新考勤和新薪资。
- `disabled`：停用，不参与新考勤和新薪资，保留历史。
- `resigned`：离职，不参与新考勤和新薪资，保留历史。

#### `attendanceRuleId`

- 名称：当前考勤规则 ID
- 类型：`number` / `string`
- 是否必填：是
- 关联：`AttendanceRule.id`
- 约束：一个员工当前只能关联一个考勤规则
- 用途：员工当前规则引用

#### `salaryType`

- 名称：计薪方式
- 类型：`enum`
- 是否必填：是
- 枚举值：
  - `fixed`
  - `hourly`
- 约束：固定工资和时薪二选一

#### `hourlyRate`

- 名称：时薪金额
- 类型：`number`
- 是否必填：`salaryType = hourly` 时必填
- 校验：大于等于 0
- 用途：时薪员工薪资核算

#### `fixedSalary`

- 名称：固定工资金额
- 类型：`number`
- 是否必填：`salaryType = fixed` 时必填
- 校验：大于等于 0
- 用途：固定工资员工自然月薪资核算

#### `currency`

- 名称：币种
- 类型：`enum`
- 是否必填：是
- 枚举值：
  - `THB`
  - `USD`
  - `MYR`
  - `IDR`

#### `photo`

- 名称：头像/照片
- 类型：`string | null`
- 是否必填：否
- 建议：后端保存图片 URL，不保存 base64 大字段

#### `isDeleted`

- 名称：是否软删除
- 类型：`boolean`
- 是否必填：是
- 默认值：`false`
- 用途：软删除标记
- 说明：可与 `status = disabled` 配合使用

#### `createdAt`

- 名称：创建时间
- 类型：`datetime`
- 生成方式：后端生成

#### `updatedAt`

- 名称：更新时间
- 类型：`datetime`
- 生成方式：后端生成

---

## 3.2 EmployeeAttendanceRuleHistory 员工考勤规则历史表

### 3.2.1 表定位

员工更换考勤规则时，必须保留历史规则依据。历史考勤和历史薪资应按当时有效规则追溯。

### 3.2.2 字段定义

#### `id`

- 历史记录 ID
- 类型：`number` / `bigint`
- 后端生成
- 必填、唯一

#### `employeeId`

- 员工 ID
- 关联：`Employee.id`
- 必填

#### `attendanceRuleId`

- 考勤规则 ID
- 关联：`AttendanceRule.id`
- 必填

#### `effectiveStartDate`

- 生效开始日期
- 类型：`date`
- 必填

#### `effectiveEndDate`

- 生效结束日期
- 类型：`date | null`
- 可为空
- 当前有效规则历史记录中可为空

#### `createdAt`

- 创建时间
- 后端生成

#### `createdBy`

- 操作人
- 可选
- 后续接权限系统时使用

### 3.2.3 约束规则

- 同一员工在同一天只能有一条有效规则。
- 同一员工的规则历史时间段不能重叠。
- 当前有效记录的 `effectiveEndDate` 可以为空。
- 更换规则时需要关闭旧历史、创建新历史、更新员工当前规则。

## 4. 接口设计

## 4.1 获取员工列表

### 用途

员工管理页列表展示和搜索筛选。

### 查询参数

- `keyword`：搜索姓名、员工编号、手机号、岗位、部门/区域
- `status`：员工状态
- `country`：国家/地区
- `salaryType`：计薪方式
- `attendanceRuleId`：当前考勤规则
- `includeInactive`：是否包含停用/离职
- `page`
- `pageSize`

### 返回字段

- `id`
- `employeeNo`
- `name`
- `gender`
- `country`
- `phone`
- `role`
- `dept`
- `joinDate`
- `status`
- `attendanceRuleId`
- `attendanceRuleName`
- `salaryType`
- `hourlyRate`
- `fixedSalary`
- `currency`
- `photo`
- `isDeleted`

### 业务规则

- 默认不返回停用/离职员工。
- `includeInactive = true` 时返回停用/离职员工。
- 离职/停用员工可在历史查询中被引用。

---

## 4.2 获取员工详情

### 输入

- `employeeId`

### 返回

- 员工主数据
- 当前考勤规则信息
- 当前薪资基础信息
- 规则历史摘要

### 业务规则

- 停用/离职员工仍可查看详情。
- 历史数据不因软删除丢失。

---

## 4.3 新增员工

### 输入字段

- `name`
- `gender`
- `country`
- `phone`
- `role`
- `dept`
- `joinDate`
- `status`
- `attendanceRuleId`
- `ruleEffectiveStartDate`
- `salaryType`
- `hourlyRate`
- `fixedSalary`
- `currency`
- `photo`

### 后端自动处理

- 自动生成 `id`
- 自动生成 `employeeNo`
- 创建 `Employee`
- 创建第一条 `EmployeeAttendanceRuleHistory`

### 校验规则

- 姓名必填。
- 手机号必填。
- 性别必填。
- 国家必填。
- 岗位必填。
- 入职日期必填。
- 当前考勤规则必填。
- 规则生效日期必填。
- 计薪方式必填。
- `salaryType = hourly` 时，`hourlyRate` 必填。
- `salaryType = fixed` 时，`fixedSalary` 必填。
- 固定工资和时薪不能同时生效。
- 当前考勤规则必须存在。

### 返回

- 创建后的员工详情。

---

## 4.4 编辑员工

### 可编辑字段

- `name`
- `gender`
- `country`
- `phone`
- `role`
- `dept`
- `joinDate`
- `status`
- `attendanceRuleId`
- `ruleEffectiveStartDate`
- `salaryType`
- `hourlyRate`
- `fixedSalary`
- `currency`
- `photo`

### 不可编辑字段

- `id`
- `employeeNo`

### 修改考勤规则

如果 `attendanceRuleId` 发生变化：

1. 必须传入新规则生效日期。
2. 关闭旧规则历史。
3. 创建新规则历史。
4. 更新 `Employee.attendanceRuleId`。
5. 不覆盖历史考勤和历史薪资使用过的旧规则记录。

### 修改计薪方式

- `salaryType = hourly` 时，只使用 `hourlyRate`。
- `salaryType = fixed` 时，只使用 `fixedSalary`。
- 固定工资和时薪不能同时生效。

### 返回

- 编辑后的员工详情。

---

## 4.5 停用/离职员工

### 用途

替代物理删除员工。

### 输入

- `employeeId`
- `targetStatus`
  - `disabled`
  - `resigned`
- `reason`，可选

### 后端处理

- 将员工状态改为 `disabled` 或 `resigned`。
- 根据实现选择是否设置 `isDeleted = true`。
- 不删除员工主数据。
- 不删除历史考勤。
- 不删除历史薪资。
- 不删除规则历史。

### 业务规则

- 停用/离职员工不参与新考勤。
- 停用/离职员工不参与新薪资。
- 停用/离职员工仍可出现在历史考勤和历史薪资查询中。

---

## 4.6 查询员工规则历史

### 输入

- `employeeId`

### 返回字段

- `id`
- `employeeId`
- `attendanceRuleId`
- `attendanceRuleName`
- `effectiveStartDate`
- `effectiveEndDate`
- `createdAt`
- `createdBy`

### 业务规则

- 返回当前规则和历史规则。
- 支持考勤计算按日期匹配历史规则。

## 5. 后端校验规则汇总

### 5.1 员工基础校验

- 员工编号由后端自动生成，且唯一。
- 姓名必填。
- 手机号必填。
- 性别必填。
- 国家必填。
- 岗位必填。
- 入职日期必填。
- 员工状态必填。
- 币种必填。

### 5.2 考勤规则关联校验

- 当前考勤规则必填。
- 当前考勤规则必须存在。
- 员工当前只能关联一条考勤规则。
- 员工规则历史时间段不能重叠。
- 新增员工时必须创建第一条规则历史。
- 修改规则时必须创建新的规则历史。

### 5.3 计薪方式校验

- 计薪方式必填。
- 固定工资和时薪只能二选一。
- `salaryType = hourly` 时，`hourlyRate` 必填且大于等于 0。
- `salaryType = fixed` 时，`fixedSalary` 必填且大于等于 0。

### 5.4 删除/停用校验

- 不允许物理删除员工。
- 删除操作必须转换为停用或离职。
- 停用/离职员工保留历史考勤、历史薪资和规则历史。

---

## 6. 状态业务规则

## 6.1 active 在职

- 参与新考勤。
- 参与新薪资。
- 可编辑资料。
- 可修改考勤规则。
- 可修改计薪方式。

## 6.2 probation 试用

- 参与新考勤。
- 参与新薪资。
- 可编辑资料。
- 可修改考勤规则。
- 可修改计薪方式。

## 6.3 on_leave 休假

- 员工仍有效。
- 考勤中可显示为休假。
- 薪资处理按后续薪资规则。
- 可编辑资料。

## 6.4 disabled 停用

- 不参与新考勤。
- 不参与新薪资。
- 保留历史考勤。
- 保留历史薪资。
- 保留规则历史。

## 6.5 resigned 离职

- 不参与新考勤。
- 不参与新薪资。
- 保留历史考勤。
- 保留历史薪资。
- 保留规则历史。

---

## 7. 后续接口对接注意事项

- 员工列表接口默认过滤停用/离职员工。
- 历史考勤和历史薪资接口不能因为员工停用/离职而丢失员工信息。
- 考勤计算接口应通过 `EmployeeAttendanceRuleHistory` 按日期匹配规则。
- 薪资核算接口应读取员工当期有效的计薪方式和薪资基础信息。
- 如果员工计薪方式后续也需要历史追溯，应新增薪资配置历史表；本次员工管理设计暂不展开。

---

## 8. 验收标准

- 后端能自动生成唯一员工编号。
- 员工主数据包含手机号。
- 员工状态支持在职、休假、试用、停用、离职。
- 员工删除采用停用/软删除，不物理删除。
- 停用/离职员工不参与新考勤和新薪资。
- 停用/离职员工可继续出现在历史考勤和历史薪资中。
- 员工当前只能关联一个考勤规则。
- 员工规则变更历史可追溯。
- 固定工资和时薪二选一。
- 新增员工时创建第一条规则历史。
- 修改员工考勤规则时生成新的规则历史。
